## 1. Python Core

### Основы

**Интерпретируемость**

Python называют интерпретируемым языком, но внутри всё хитрее. При запуске код сначала компилируется в **байт-код** — низкоуровневые инструкции для виртуальной машины Python. Байт-код сохраняется в папке `__pycache__` с расширением `.pyc`. Затем виртуальная машина выполняет эти инструкции построчно. Таким образом, Python представляет собой гибрид: компиляция в байт-код + интерпретация байт-кода. Компиляция остаётся незаметной, потому что отсутствует отдельный этап сборки исполняемого файла — всё происходит автоматически во время запуска.

Установка Python — это установка виртуальной машины (PVM — Python Virtual Machine) вместе с компилятором и стандартной библиотекой. Именно виртуальная машина выполняет байт-код. Благодаря такой архитектуре один и тот же код работает на Windows, Linux и macOS без переписывания: под каждую платформу существует своя реализация виртуальной машины, понимающая единый формат байт-кода.

**PEP (Python Enhancement Proposal)**

Документы, в которых предлагаются и описываются изменения языка, а также устанавливаются стандарты (стиль кода, аннотации типов и т.д.). Важные PEP:
- **PEP 8** — руководство по стилю кода.
- **PEP 20** — "Дзен Python".
- **PEP 257** — соглашения по docstrings.
- **PEP 484** — аннотации типов.
- **PEP 572** — оператор `:=`.
- **PEP 634-636** — структурный паттерн-матчинг (`match/case`).

**Версии Python (ключевые изменения)**

- **Python 2 → 3 (2008):** print как функция, Unicode по умолчанию, деление `/` теперь возвращает float (раньше был int), `//` для целочисленного деления.  
- **3.6 (2016):** f-строки, аннотации типов для переменных.  
- **3.7 (2018):** `dataclasses`, `async`/`await` стали ключевыми словами.  
- **3.8 (2019):** `:=` (walrus operator), позиционные и именованные аргументы (подробнее в разделе "Функции").  
- **3.9 (2020):** аннотации типов для встроенных коллекций (`list[int]` вместо `List[int]`), операторы объединения словарей (`|`).  
- **3.10 (2021):** паттерн-матчинг (`match/case`), более точные сообщения об ошибках.  
- **3.11 (2022):** значительное ускорение (до 60% быстрее 3.10), группы исключений (`ExceptionGroup`).  
- **3.12 (2023):** еще больше оптимизаций, поддержка f-строк внутри f-строк, улучшенный профилировщик.  
- **3.13 (2024):** улучшенная производительность, экспериментальный режим без GIL.

**Реализации Python**

Python — это спецификация языка, у которой есть несколько реализаций:

- **CPython** — эталонная реализация на C. Самая распространённая, именно её скачивают с python.org. Все C-расширения (numpy, pandas) пишутся под неё. Содержит GIL (см. подраздел "Управление памятью и сборщик мусора").
- **PyPy** — реализация на Python (с JIT-компиляцией). Часто быстрее CPython, стремится к полной совместимости. Содержит GIL.
- **Jython** — реализация на Java, работает на JVM. Позволяет использовать Java-классы из Python. Не содержит GIL.
- **IronPython** — реализация на C# для платформы .NET. Не содержит GIL.
- **MicroPython** — облегчённая версия для микроконтроллеров и встраиваемых систем. Не содержит GIL (нет многопоточности).

### Управление памятью и сборщик мусора (GC, GIL)

**Подсчёт ссылок (Reference Counting)** — основной механизм управления памятью в Python. Каждый объект хранит счётчик, сколько раз на него ссылаются. Когда счётчик падает до нуля, объект удаляется немедленно. Это происходит **синхронно** и не требует отдельного потока.

    a = []    # счётчик = 1
    b = a     # счётчик = 2
    del a     # счётчик = 1
    del b     # счётчик = 0 → объект удалён

Проблема: подсчёт ссылок не работает с циклическими ссылками (объекты ссылаются друг на друга, но извне недоступны). Их счётчики никогда не обнулятся.

    class Node:
        def __init__(self):
            self.ref = None

    a = Node()
    b = Node()
    a.ref = b  # взаимные ссылки
    b.ref = a
    del a
    del b  # объекты остались в памяти (цикл)

**GC (Garbage Collector)** — сборщик мусора, который приходит на помощь. Он периодически запускается, находит циклические ссылки и удаляет их.

**Как работает GC:**
- Запускается автоматически, когда количество вновь созданных объектов минус количество удалённых превышает порог (по умолчанию 700)
- Делит объекты на поколения (0, 1, 2). Новые объекты попадают в поколение 0. Если объект переживает сборку, он перемещается в старшее поколение. Старшие поколения проверяются реже.
- При запуске проходит по графу объектов, отмечает достижимые, удаляет недостижимые циклы

**Как управлять:**
- `gc.disable()` — отключить автоматическую сборку
- `gc.collect()` — запустить сборку вручную
- `gc.set_threshold(threshold0, threshold1, threshold2)` — настроить пороги поколений
- `gc.get_objects()` — посмотреть все объекты, отслеживаемые GC

**GIL (Global Interpreter Lock)**

Блокировка в CPython (и некоторых других реализациях), разрешающая выполнение только одного потока в любой момент времени в рамках одного процесса. Это упрощает управление памятью (подсчёт ссылок становится потокобезопасным без множества мелких блокировок), но ограничивает параллелизм для CPU-bound задач. Для I/O-bound задач GIL освобождается во время системных вызовов, поэтому потоки остаются эффективными при работе с сетью и диском. В реализациях на JVM (Jython) и .NET (IronPython) GIL отсутствует, так как они используют потоковую модель своих платформ.

**Связь GIL и GC:** когда GC запускается, он захватывает GIL и не отпускает до завершения. В это время все другие потоки с Python-кодом блокируются. Это может вызывать незапланированные паузы (stop-the-world) в многопоточных приложениях. В асинхронном коде (один поток) паузы GC тоже есть, но проблема остановки других потоков не возникает.

### Типы данных

**Изменяемые (Mutable):** `list`, `set`, `dict`, пользовательские классы.  
**Неизменяемые (Immutable):** `int`, `float`, `str`, `tuple`, `frozenset`.

**Абстрактные базовые классы из `collections.abc`**  
Описывают интерфейсы, а не реализации:

- **`Sequence`** — контракт для последовательностей с индексацией (`__getitem__`, `__len__`).  
  *Неизменяемые реализации:* `tuple`, `str`.  
  *Изменяемые реализации (`MutableSequence`):* `list`.

- **`Mapping`** — контракт для словарей с доступом по ключу (`__getitem__`, `__len__`, `__iter__`).  
  *Изменяемые реализации (`MutableMapping`):* `dict`.

- **`Container`** — контракт для проверки вхождения (`__contains__`).  
  *Реализации:* `list`, `set`, `dict`, `str` и др.

- **`Set`** — контракт для множеств (`__contains__`, `__iter__`, `__len__`).  
  *Неизменяемые реализации:* `frozenset`.  
  *Изменяемые реализации (`MutableSet`):* `set`.

**Срезы (Slicing)**  
Механизм получения подстрок/подсписков: `[start:stop:step]`. Работает благодаря `__getitem__`. Пример: `"hello"[::-1]` → `"olleh"`.

**Особенности numpy.array и pandas.DataFrame**

**numpy.array** — изменяемый объект: можно менять элементы по индексам и присваивать в срезы. Однако операции вроде `np.append()` возвращают **новый** массив, не изменяя исходный.

    import numpy as np

    arr = np.array([1, 2, 3])

    # Изменение существующего массива — работает
    arr[0] = 999
    arr[:] = arr * 2  # присваивание в срез

    # "Добавление" элемента — создаёт новый массив
    new_arr = np.append(arr, 4)  # arr остался прежним

**Почему так сделано:** массивы numpy хранят данные в непрерывном блоке памяти фиксированного размера. Добавление элемента требует полного копирования (O(n)). Частые `append` — антипаттерн. Правильный подход: собрать данные в список (амортизированное O(1)), затем преобразовать в numpy.

    # Хорошо
    lst = []
    for i in range(100000):
        lst.append(i)
    arr = np.array(lst)

**pandas.DataFrame** — тоже изменяемый. При передаче в функцию изменения применяются к оригиналу, если не создана явная копия (`.copy()`). Многие операции (например, фильтрация) возвращают новые объекты.


### Дженерики (Generics)

Позволяют параметризовать класс или функцию типом, который указывается при использовании. Существуют только для статической проверки (mypy, IDE). В рантайме информация о типе стирается.

**TypeVar** — объявление переменной типа.

    from typing import TypeVar

    T = TypeVar('T')                     # любой тип
    Num = TypeVar('Num', int, float)     # только int или float
    A = TypeVar('A', bound=Animal)       # только Animal и его наследники

**Generic[T]** — класс, параметризованный типом T. Без `Generic[T]` mypy не свяжет тип `T` в методах с типом, указанным при создании экземпляра.

    from typing import Generic, List

    class Stack(Generic[T]):
        def __init__(self) -> None:
            self._items: List[T] = []

        def push(self, item: T) -> None:
            self._items.append(item)

        def pop(self) -> T:
            return self._items.pop()

    int_stack = Stack[int]()  # T = int
    int_stack.push(1)
    print(int_stack.pop())    # 1

**Ограничения:**
- Нельзя использовать в `isinstance()` (в рантайме нет информации о типе)
- В рантайме `Stack[int]` и `Stack[str]` — один и тот же класс

**Когда нужны:** класс или функция должны работать с разными типами, но сохранять типобезопасность внутри.

### Дженерики в рантайме

Если дженерик-параметр нужен в рантайме (например, для валидации), значение типа приходится передавать явно.

**Вариант 1: Pydantic (автоматическая валидация)**

    from pydantic import BaseModel, ValidationError

    class Box(BaseModel, Generic[T]):
        item: T

    # При Box[int] T становится int → item: int
    box_int = Box[int](item="5")   # Pydantic приводит "5" к int
    print(box_int.item)            # 5

    # При Box[str] T становится str → item: str
    box_str = Box[str](item=42)    # Pydantic приводит 42 к str
    print(box_str.item)            # "42"

    # Ошибка: "not a number" нельзя привести к int
    try:
        Box[int](item="not a number")
    except ValidationError as e:
        print(e)

**Суть дженерика:** вместо `T` подставляется конкретный тип (`int`, `str`).  
**Механика Pydantic:** автоматическое приведение типов (str → int, int → str) — относится ко всем моделям, не только к дженерикам.

**Вариант 2: Ручная передача типа**

    class BaseConsumer(Generic[T]):
        def __init__(self, message_cls: Type[T]):
            self.message_cls = message_cls

    class SSEConsumer(BaseConsumer[SSEMessageSchema]):
        def __init__(self):
            super().__init__(SSEMessageSchema)  # обязательно

**Фабричная функция (без дублирования типа):**

    def make_consumer(message_cls: Type[T]) -> BaseConsumer[T]:
        return BaseConsumer[T](message_cls)

    c = make_consumer(SSEMessageSchema)  # тип выводится автоматически


### Функции и области видимости

#### Передача аргументов

В Python функция получает в качестве аргумента не сам объект, а ссылку на него. При этом внешняя переменная продолжает ссылаться на тот же объект, даже если внутри функции локальной переменной присвоили ссылку на другой объект.

**Что это даёт на практике:**

- **Изменяемые объекты** (списки, словари, экземпляры классов) — функция может зайти по ссылке и поменять содержимое. Изменения будут видны снаружи.
- **Неизменяемые объекты** (числа, строки, кортежи) — содержимое поменять нельзя. Любая операция создаёт новый объект, и локальная переменная начинает ссылаться на него. Внешняя переменная продолжает ссылаться на старый объект.

Пример:

    def process(data, value):
        data.append(value)      # меняем содержимое списка (изменяемый)
        value = value + 1       # создаём новое число (неизменяемый)

    my_list = [1, 2]
    my_num = 5

    process(my_list, my_num)

    print(my_list)  # [1, 2, 5]  — список изменился
    print(my_num)   # 5           — число не изменилось

**Позиционные и именованные аргументы**

- **`/`** — параметры **до** слеша: только позиционные (нельзя передать по имени). Их можно переименовывать без риска сломать код.
- **`*`** — разделитель. Параметры **после** звёздочки: только именованные (нельзя передать позиционно). Лишние позиционные аргументы вызывают ошибку.
- **`*args`** — собирает лишние позиционные аргументы в кортеж. Параметры после `*args` — только именованные.
- **`**kwargs`** — собирает лишние именованные аргументы в словарь. Всегда последний.

Примеры:

**1. Только разделитель `*`**

    def func(a, *, b):
        pass

    func(1, b=2)        # ✅
    func(1, 2, b=3)     # ❌ лишний позиционный аргумент 2

**2. Сборщик `*args`**

    def func(a, *args, b):
        pass

    func(1, 2, 3, b=4)  # ✅ args = (2, 3)
    func(1, b=2)        # ✅ args = ()
    func(1, 2, 3, 4)    # ❌ b не передан

**Полный синтаксис:**

    def func(a, b, /, c, d, *args, e, f, **kwargs):
        pass

    # a, b — только позиционные
    # c, d — можно и позиционно, и по имени
    # args — кортеж с лишними позиционными аргументами
    # e, f — только по имени
    # kwargs — словарь с лишними именованными аргументами

**Замыкание (Closure)**

Функция, запоминающая состояние переменных из внешней области видимости даже после выхода из нее:

    def make_adder(x):
        def adder(y):
            return x + y  # x запоминается (замыкание)
        return adder

    add5 = make_adder(5)
    print(add5(3))  # 8


**global** — позволяет обращаться к переменной, объявленной на уровне модуля (глобальной).

    x = 10  # глобальная

    def func():
        global x
        x = 20  # меняем глобальную переменную

    func()
    print(x)  # 20

Без `global` внутри функции создалась бы новая локальная переменная x.

**nonlocal** — работает для вложенных функций. Позволяет менять переменную из внешней функции (не глобальную).

    def outer():
        x = 10  # переменная внешней функции
        def inner():
            nonlocal x
            x = 20  # меняем x из outer
        inner()
        print(x)  # 20


**Зачем нужны:** без `nonlocal` в `inner` создалась бы своя локальная x, не влияя на внешнюю. Без них функции не могут изменять переменные из внешних областей видимости (кроме изменяемых объектов — там меняется содержимое, а не сама переменная).

#### Асинхронные функции и корутины

- **Асинхронная функция** — функция, определённая через `async def`. При вызове возвращает **сопрограмму**.
- **Сопрограмма (coroutine)** — объект, который можно выполнить через `await` — тогда выполнение приостанавливается до её завершения.
- **Future** — объект, представляющий результат, который появится в будущем. `await future` завершится, когда кто-то вызовет `future.set_result(value)`.
- **Задача (Task)** — наследник Future, обёртка над сопрограммой, созданная с помощью `create_task()`. Задача сразу отправляет сопрограмму в цикл событий, где она выполняется асинхронно с другим кодом, даже без команды `await task`, с помощью которой результат можно получить когда угодно позже.
- **awaitable** — объект, который можно использовать с `await` (корутина, задача, Future).


**Пример в коде:**

    import asyncio

    async def fetch_data_coro():
        await asyncio.sleep(1)
        return "data"

    async def fetch_data_task(future):
        await asyncio.sleep(1)
        # Устанавливаем результат для Future
        future.set_result("data")
        await asyncio.sleep(0.5)
        return "data"

    future = asyncio.Future()  # Future просто создаётся
    coro = fetch_data_coro()   # Сопрограмма создана, но ещё не запущена
    task = asyncio.create_task(fetch_data_task(future))  # Задача: создана и сразу начинает выполняться в фоне
    await asyncio.sleep(0.5)  # Можно делать другие дела

    # Дождаться результата разными способами
    result1 = await future        # ждём, пока кто-то вызовет future.set_result("data")
    result2 = await task          # задача уже выполняется, здесь просто ждём её завершения
    result3 = await coro          # сопрограмма только сейчас начнёт выполняться и завершится здесь
    print(result1, result2, result3)  # "data" "data" "data"

#### Декораторы

Функции, расширяющие поведение другой функции. Выполняются сразу после определения. Декораторы могут передать `*args, **kwargs` в исходную функцию, а могут изменить или дополнить их.

- **Простой декоратор** — принимает функцию/класс, возвращает функцию/класс.
- **Фабрика декоратора** — принимает параметры, возвращает декоратор.

**Синхронные декораторы**

    def timer_and_session(func):
        def wrapper(*args, **kwargs):
            start = time.time()
            with Session() as session:
                result = func(*args, session, **kwargs)
            print(f"{func.__name__} took {time.time()-start}s")
            return result
        return wrapper

    @timer_and_session
    def add_message(session, msg):
        session.add(msg)

    add_message("Hello")

**Асинхронные декораторы**

    def async_timer_and_session(func):
        async def wrapper(*args, **kwargs):
            start = time.time()
            async with AsyncSession() as session:
                result = await func(*args, session, **kwargs)
            print(f"{func.__name__} took {time.time()-start}s")
            return result
        return wrapper

    @async_timer_and_session
    async def add_message_async(session, msg):
        await session.add(msg)

    await add_message_async("Hello")

**Фабрика декоратора (с параметрами)**

    def repeat(n):
        def decorator(func):
            def wrapper(*args, **kwargs):
                for _ in range(n):
                    result = func(*args, **kwargs)
                return result
            return wrapper
        return decorator

    @repeat(3)
    def greet(name):
        print(f"Hello, {name}!")

    greet("Alex")  # выведет 3 раза

**Декоратор класса**

    def add_method(cls):
        cls.new_method = lambda self: "Hello"
        return cls

    @add_method
    class MyClass:
        pass

    obj = MyClass()
    print(obj.new_method())  # "Hello"

#### functools — утилиты для функций

`partial` — фиксирует часть аргументов функции, создавая новую с меньшим числом параметров.

    from functools import partial

    def power(base, exponent):
        return base ** exponent

    square = partial(power, exponent=2)
    cube = partial(power, exponent=3)

    print(square(5))  # 25
    print(cube(5))    # 125

`lru_cache` — кэширует результаты вызовов функции. При повторном вызове с теми же аргументами возвращает закэшированное значение, не выполняя функцию заново.

    from functools import lru_cache

    @lru_cache(maxsize=128)
    def fibonacci(n):
        if n < 2:
            return n
        return fibonacci(n-1) + fibonacci(n-2)

`reduce` — последовательно применяет функцию к элементам, сводя последовательность к одному значению.

    from functools import reduce
    from operator import mul

    numbers = [1, 2, 3, 4, 5]
    product = reduce(mul, numbers)  # 1*2*3*4*5 = 120

`wraps` — используется в декораторах, чтобы сохранить имя, docstring и другие метаданные оригинальной функции.

    from functools import wraps

    def my_decorator(func):
        @wraps(func)  # без wraps имя станет 'wrapper'
        def wrapper(*args, **kwargs):
            print("Что-то делаем до")
            result = func(*args, **kwargs)
            print("Что-то делаем после")
            return result
        return wrapper

    @my_decorator
    def hello():
        """Документация функции hello"""
        print("Hello")

    print(hello.__name__)  # 'hello' (без wraps было бы 'wrapper')
    print(hello.__doc__)   # 'Документация функции hello'

`singledispatch` — создаёт обобщённую функцию, поведение которой зависит от типа первого аргумента (перегрузка по типу).

    from functools import singledispatch

    @singledispatch
    def process(data):
        return f"Обработка неизвестного типа: {type(data).__name__}"

    @process.register(int)
    def _(data):
        return f"Обработка целого числа: {data * 2}"

    @process.register(str)
    def _(data):
        return f"Обработка строки: {data.upper()}"

    @process.register(list)
    def _(data):
        return f"Обработка списка: {sum(data)}"

    print(process(10))       # Обработка целого числа: 20
    print(process("hello"))  # Обработка строки: HELLO
    print(process([1,2,3]))  # Обработка списка: 6
    print(process(3.14))     # Обработка неизвестного типа: float

#### @overload

Используется для подсказки типов — указывает несколько возможных сигнатур функции. Сама логика пишется один раз в реализации без `@overload` (или с ним, но тогда он игнорируется).

    from typing import overload

    @overload
    def process(data: int) -> str: ...

    @overload
    def process(data: str) -> list[str]: ...

    def process(data):
        if isinstance(data, int):
            return str(data)
        return data.split()

`@overload` не создаёт несколько версий функции. Это подсказка для системы типов (mypy, Pyright, IDE), чтобы проверки типов работали корректно. Если убрать все `@overload`, код будет выполняться так же — просто IDE перестанет понимать, какие типы возвращаются для разных входных данных.

### Итераторы, итерируемые объекты, генераторы и включения

**Итерируемый объект (iterable)** — объект, который можно передать в цикл `for`. У него есть метод `__iter__()`, возвращающий **итератор**. Примеры: список, строка, словарь, множество.

**Итератор (iterator)** — объект, который умеет отдавать элементы по одному. У него есть метод `__next__()`, возвращающий следующий элемент или бросающий `StopIteration`. Также имеет метод `__iter__()`, возвращающий самого себя (чтобы итератор можно было использовать в `for`).

**Генератор (generator)** — частный случай итератора. Создаётся либо функцией с `yield`, либо генераторным выражением `(x for x in ...)`. Работает так же, как итератор, но элементы не хранит, а вычисляет на лету.

**Важное соотношение:**
- Любой итератор является итерируемым объектом (у него есть `__iter__`)
- Не любой итерируемый объект является итератором (у списка нет `__next__`)
- Генератор — это всегда итератор (и, соответственно, итерируемый объект)

**Генераторные выражения**  
`(x for x in range(10))` — ленивый генератор, созданный через выражение. Не занимает память целиком, вычисляет элементы по одному по требованию.

**Включения (comprehensions)** — жадные конструкции, сразу создающие готовую коллекцию в памяти:

    # Списковое включение
    lst = [x for x in range(10)]

    # Множественное включение
    st = {x for x in range(10)}

    # Словарное включение
    dct = {x: x**2 for x in range(10)}

**Пример: итератор и итерируемый объект**

    class Countdown:
        """Итерируемый объект, возвращающий новый итератор"""
        def __init__(self, n):
            self.n = n

        def __iter__(self):
            return CountdownIterator(self.n)

    class CountdownIterator:
        """Итератор для обратного отсчёта"""
        def __init__(self, n):
            self.current = n

        def __iter__(self):
            return self

        def __next__(self):
            if self.current <= 0:
                raise StopIteration
            self.current -= 1
            return self.current

    # Использование
    counter = Countdown(3)        # итерируемый объект
    for x in counter:             # counter.__iter__() даёт итератор
        print(x)                   # 2, 1, 0

    # Генератор делает то же самое короче
    def countdown(n):
        while n > 0:
            yield n
            n -= 1

    for x in countdown(3):        # countdown(3) возвращает генератор
        print(x)                   # 2, 1, 0

**Как `for` работает внутри**

Когда вы пишете:

    for элемент in объект:
        тело_цикла

Python делает примерно следующее:

    итератор = iter(объект)          # вызывает объект.__iter__()
    while True:
        try:
            элемент = next(итератор) # вызывает итератор.__next__()
            тело_цикла
        except StopIteration:
            break

Здесь `объект` — это любой итерируемый объект (список, строка, генератор и т.д.).

**Быстрый поиск в генераторе**

    next((x for x in gen if условие(x)), None)

Находит первый элемент в генераторе, удовлетворяющий условию, и не вычисляет остальные. Второй аргумент `None` возвращается, если ничего не найдено (иначе `next()` бросит `StopIteration`).

### Асинхронные итераторы и генераторы

Работают по тому же принципу, что и синхронные, но с `async`/`await`. Вместо `__iter__` и `__next__` используются `__aiter__` и `__anext__`, а вместо `for` — `async for`.

**Асинхронный итератор** — объект с методами:
- `__aiter__()` — возвращает сам итератор
- `__anext__()` — возвращает awaitable (корутину), который при ожидании даёт следующий элемент или бросает `StopAsyncIteration`


    class AsyncRange:
        def __init__(self, limit):
            self.limit = limit
            self.current = 0

        def __aiter__(self):
            return self

        async def __anext__(self):
            if self.current >= self.limit:
                raise StopAsyncIteration
            self.current += 1
            return self.current

    async def main():
        async for num in AsyncRange(5):
            print(num)

**Асинхронный генератор** — частный случай асинхронного итератора. Функция с `async def` и `yield`. Python сам создаёт объект с нужными методами (`__aiter__` и `__anext__`).


    async def async_range(limit):
        for i in range(limit):
            yield i
            await asyncio.sleep(0.1)

    async def main():
        async for num in async_range(5):
            print(num)

**Назначение:** асинхронные генераторы нужны для потоковой передачи данных, когда элементы поступают не мгновенно, а с задержками (чанки из БД, ответы от API, строки из лога по сети). Позволяют обрабатывать данные по мере поступления, не блокируя событийный цикл.

### ООП (Объектно-Ориентированное Программирование)

**Принципы ООП:**

- **Инкапсуляция** — сокрытие внутреннего состояния объекта и предоставление доступа к нему только через методы. В Python это реализовано на уровне соглашений, а не жёсткой изоляции:
  - `_protected` — соглашение: "не трогай снаружи" (но технически доступ есть)
  - `__private` — имя искажается (name mangling): `__age` внутри класса превращается в `_ClassName__age`, что усложняет доступ, но не запрещает его полностью

        class User:
            def __init__(self):
                self.__age = 30

        u = User()
        print(u.__age)               # AttributeError
        print(u._User__age)          # 30 — доступ есть, но неочевидно

  - Полноценной приватности, как в Java или C++, в Python нет

- **Наследование** — создание нового класса на основе существующего. Дочерний класс получает методы и атрибуты родителя, может их переопределять или дополнять.

- **Полиморфизм** — способность объектов разных классов по-разному реализовывать один и тот же метод. В Python достигается через:
  - переопределение методов в наследниках (как в классическом ООП)
  - утиную типизацию: объект считается подходящим, если у него есть нужный метод, независимо от того, от какого класса он унаследован (в отличие от Java/C++, где требуется явное наследование или реализация интерфейса)

- **Абстракция** — выделение общего интерфейса, скрытие деталей реализации. В Python реализуется через абстрактные классы (`ABC`, `@abstractmethod`) или протоколы (`Protocol`).

**MRO (Method Resolution Order)**  
Порядок разрешения методов при множественном наследовании. В Python используется алгоритм C3-линеаризации.  
[python.org](https://www.python.org/download/releases/2.3/mro/)

    class A: pass
    class B(A): pass  # B наследует A
    class C(A): pass  # C наследует A
    class D(B, C): pass  # D наследует B и C
    print(D.__mro__)  # (D, B, C, A, object) — порядок поиска методов

**Магические методы (Dunder methods)**  
Методы с двумя подчеркиваниями: `__init__`, `__new__`, `__getitem__`, `__call__`, `__enter__/__exit__` и др. Определяют поведение для встроенных операций.

**__new__ и __init__**  
- `__new__(cls, ...)` — создает экземпляр (возвращает новый объект). Первый аргумент — класс (`cls`). Вызывается до `__init__`.  
- `__init__(self, ...)` — инициализирует созданный экземпляр. Ничего не возвращает (должен возвращать `None`).

"**Singleton** — гарантирует один экземпляр класса. *Пример: логгер, кэш, пул соединений.*

    class Singleton:
        _instance = None
        def __new__(cls):
            if cls._instance is None:
                cls._instance = super().__new__(cls)
            return cls._instance

### Абстрактные классы и интерфейсы

Сравнение: Обычный класс, Абстрактный класс, Интерфейс

| Характеристика | Обычный класс | Абстрактный класс | Интерфейс |
|---|---|---|---|
| Создание экземпляра | ✅ можно | ⛔ нельзя | ⛔ нельзя |
| Наличие реализации методов | ✅ может быть | ✅ может быть (обычные методы) | ❌ только сигнатуры (все методы абстрактные) |
| Наследник обязан реализовать метод | ❌ не обязан | ✅ обязан (если метод помечен `@abstractmethod`) | ✅ обязан (все методы) |
| Контроль на этапе создания наследника | ❌ нет (ошибка только при вызове непереопределённого метода) | ✅ есть: ошибка при создании, если не реализованы все методы, помеченные `@abstractmethod` (часть методов) | ✅ есть: ошибка при создании, если не реализованы все методы, помеченные `@abstractmethod` (все методы) |
| Может содержать переменные/свойства | ✅ да | ✅ да | ❌ нет (только методы) |
| Множественное наследование | ✅ да (но осторожно с MRO) | ✅ да | ✅ да (безопасно, т.к. нет конфликтов реализации) |
| Основное назначение | Переиспользование кода | Переиспользование кода + задание контракта | Только задание контракта |

**Важные уточнения**

**1. @abstractmethod работает только вместе с ABC**

Просто написать `@abstractmethod` недостаточно — класс **обязан** наследоваться от `ABC` (или другого класса, который уже наследуется от `ABC`):

    from abc import ABC, abstractmethod

    # Правильно
    class AbstractAnimal(ABC):
        @abstractmethod
        def make_sound(self):
            pass

    # Неправильно — @abstractmethod проигнорируется
    class BrokenAnimal:
        @abstractmethod
        def make_sound(self):
            pass

    # Этот объект создастся, хотя метод абстрактный!
    broken = BrokenAnimal()  # ✅ работает (хотя не должен)

**2. @abstractmethod vs raise NotImplementedError — это не одно и то же**

    class WithABC(ABC):
        @abstractmethod
        def must_implement(self):
            pass

    class WithError:
        def must_implement(self):
            raise NotImplementedError()

    # Разница в поведении:
    try:
        obj = WithABC()  # ❌ TypeError сразу
    except TypeError as e:
        print(f"Ошибка: {e}")

    obj = WithError()    # ✅ создаётся без проблем
    obj.must_implement() # ❌ ошибка только здесь — поздно и неочевидно

**3. Протоколы (Protocol) — только для статической проверки**

`Protocol` из `typing` не даёт ошибок в рантайме. Это подсказка для линтеров и IDE: "объект должен иметь такие-то методы". Если методы не реализованы — mypy или Pyright покажут ошибку, но код выполнится.

    from typing import Protocol

    class Drawable(Protocol):
        def draw(self) -> None: ...
        def resize(self, factor: float) -> None: ...

    def render(obj: Drawable):
        obj.draw()
        obj.resize(2.0)

    class Circle:
        def draw(self):
            print("○")
        
        def resize(self, factor):
            print(f"resize {factor}")

    render(Circle())  # ✅ работает в рантайме, хотя Circle не наследует Drawable
    # mypy ошибку покажет, только если методов не хватает

**Примеры кода с демонстрацией**

    from abc import ABC, abstractmethod

    # ----- 1. Обычный базовый класс -----
    class RegularUser:
        def __init__(self, name):
            self.name = name
        
        def get_role(self):
            return "user"
        
        def get_permissions(self):
            raise NotImplementedError("Переопредели, если нужно")

    # ----- 2. Абстрактный класс (есть реализация + абстрактные методы) -----
    class AbstractUser(ABC):
        def __init__(self, name):
            self.name = name
        
        @abstractmethod
        def get_role(self):
            pass
        
        def get_permissions(self):   # готовый метод
            return ["read"]

    # ----- 3. Интерфейс через ABC (все методы абстрактные) -----
    class UserInterface(ABC):
        @abstractmethod
        def get_role(self):
            pass
        
        @abstractmethod
        def get_permissions(self):
            pass
        # нет __init__, нет готовых методов, нет свойств

    # ----- 4. Множественное наследование интерфейсов -----
    class Reader(ABC):
        @abstractmethod
        def read(self): ...

    class Writer(ABC):
        @abstractmethod
        def write(self): ...

    class FileHandler(Reader, Writer):
        def read(self):
            print("reading")
        
        def write(self):
            print("writing")

    # ----- Проверка: что работает, а что нет -----

    # 1. Обычный класс
    regular = RegularUser("John")           # ✅ создаётся
    regular.get_permissions()                # ❌ NotImplementedError (только при вызове)

    # 2. Абстрактный класс
    # abstract = AbstractUser("John")        # ⛔ TypeError! нельзя создать

    class Admin(AbstractUser):
        def get_role(self):                   # обязательно
            return "admin"
        # get_permissions уже есть

    admin = Admin("Alex")                     # ✅ работает

    # 3. Интерфейс через ABC
    # interface = UserInterface()             # ⛔ TypeError! нельзя создать

    class Guest(UserInterface):
        def get_role(self):
            return "guest"
        
        def get_permissions(self):            # обязательно всё
            return []

    guest = Guest()                           # ✅ работает

    class Broken(UserInterface):               # забыли реализовать методы
        pass

    # broken = Broken()                       # ⛔ TypeError!

    # 4. Множественное наследование интерфейсов
    handler = FileHandler()                    # ✅ работает
    handler.read()
    handler.write()

**Когда что выбирать**

| Нужно | Берём |
|---|---|
| Просто собрать общую логику для переиспользования | Обычный класс |
| Общая логика + жёсткое требование реализовать часть методов | Абстрактный класс |
| Только описать, что умеет объект (рантайм контроль) | Интерфейс |
| Только описать, что умеет объект (статическая проверка, без накладных расходов в рантайме) | Protocol |

**__slots__**

Атрибут класса, ограничивающий набор допустимых атрибутов экземпляра. Экономит память (нет `__dict__`) и ускоряет доступ к атрибутам.

    class Point:
        __slots__ = ('x', 'y')  # только эти атрибуты разрешены
        def __init__(self, x, y):
            self.x = x
            self.y = y

    p = Point(1, 2)
    p.z = 3  # AttributeError

**Наследование:**
- Если и родитель, и потомок определили `__slots__`, потомок получает слоты родителя + свои.
- Если родитель **не определил** `__slots__`, у потомка всё равно появится `__dict__`, даже если потомок определил свои слоты. Тогда ограничения на добавление атрибутов снимаются.

**Когда использовать:**
- Миллионы объектов (экономия памяти критична)
- Набор атрибутов известен заранее и не будет меняться

**Когда не использовать:**
- Нужно динамически добавлять атрибуты
- Класс участвует в сложном наследовании с классами без `__slots__`

### Слабые ссылки (weakref)

Обычная ссылка (`obj = object()`) удерживает объект в памяти. Слабая ссылка (`weakref.ref(obj)`) не препятствует удалению объекта сборщиком мусора.

**Пример: кэш, который не блокирует удаление объектов**

    import weakref

    class User:
        def __init__(self, name):
            self.name = name

    # Обычный словарь — сильные ссылки
    strong_cache = {}
    user = User("Alice")           # счётчик ссылок = 1 (user)
    strong_cache["alice"] = user   # счётчик = 2 (user + strong_cache["alice"])
    del user                       # счётчик = 1 (strong_cache["alice"])
    print(strong_cache.get("alice"))  # <User object> — объект жив

    # Слабый словарь — только слабые ссылки
    weak_cache = weakref.WeakValueDictionary()
    user = User("Bob")             # счётчик = 1 (user)
    weak_cache["bob"] = user       # слабая ссылка не увеличивает счётчик, счётчик = 1
    del user                       # счётчик = 0, объект удалён
    print(weak_cache.get("bob"))   # None — объект удалён

**Где используется:**
- **Кэши** — чтобы объекты не висели в памяти вечно
- **Решение проблемы циклических ссылок** — когда объекты ссылаются друг на друга
- **Паттерн Observer** — чтобы подписчики не мешали удалению наблюдаемого объекта

**Для классов с `__slots__`:** если нужны слабые ссылки, добавьте `'__weakref__'` в список слотов.

**Переменные класса и экземпляра**  
Переменные класса общие для всех экземпляров, переменные экземпляра — уникальные для каждого.


    class User:
        class_property = 1           # неизменяемая переменная класса
        class_list = []              # изменяемая переменная класса

        def __init__(self, name):
            self.name = name         # переменная экземпляра

    u1 = User("Alice")
    u2 = User("Bob")

    # Неизменяемая переменная класса
    print(u1.class_property)         # 1
    print(u2.class_property)         # 1
    User.class_property = 2
    print(u1.class_property)         # 2
    print(u2.class_property)         # 2

    # Присваивание через экземпляр — создаётся своя переменная экземпляра
    u1.class_property = 10
    print(u1.class_property)         # 10 (экземплярная)
    print(u2.class_property)         # 2 (классовая)
    print(User.class_property)       # 2 (классовая)

    # Изменяемая переменная класса — изменение через экземпляр влияет на класс
    u1.class_list.append(100)
    u2.class_list.append(200)
    User.class_list.append(300)

    print(u1.class_list)             # [100, 200, 300]
    print(u2.class_list)             # [100, 200, 300]

    # Присваивание через экземпляр — теперь это локальная переменная экземпляра
    u1.class_list = [111]
    print(u1.class_list)             # [111] (экземплярная)
    print(u2.class_list)             # [100, 200, 300] (классовая)

    # Перезапись классовой переменной
    User.class_list = [1000]
    print(u1.class_list)             # [111] (экземплярная)
    print(u2.class_list)             # [1000] (классовая)

    # Новый экземпляр
    u3 = User("Charlie")
    print(u3.class_property)         # 2
    print(u3.class_list)             # [1000]

**super() — подробно**

`super()` — функция, возвращающая прокси-объект, который делегирует вызовы методов согласно MRO. Принимает два необязательных аргумента:

- **Первый** — класс, **с которого начать поиск** (исключая его самого). По умолчанию — текущий класс.
- **Второй** — объект (экземпляр или класс), который будет передан как `self` (для методов) или `cls` (для классметодов) при вызове найденного метода. По умолчанию — текущий экземпляр или класс. Для статических методов второй аргумент игнорируется.

**Где можно вызывать:**
- `super()` — только внутри класса (автоматически подставляет текущий класс и self/cls)
- `super(Class)` — только для статических методов (второй аргумент не нужен)
- `super(Class, obj)` — в любом месте (obj может быть экземпляром или классом)

**Примеры вызовов:**

    class A:
        def method(self): return "A"
        @classmethod
        def clsmethod(cls): return f"A from {cls.__name__}"
        @staticmethod
        def stmethod(): return "A static"

    class B(A):
        def method(self): return "B -> " + super().method()
        @classmethod
        def clsmethod(cls): return "B -> " + super().clsmethod()
        @staticmethod
        def stmethod(): return "B static -> " + super().stmethod()

    class C(A):
        def method(self): return "C -> " + super().method()
        @classmethod
        def clsmethod(cls): return "C -> " + super().clsmethod()

    class D(B, C):
        # MRO для D: (D, B, C, A, object)
        
        def method(self):
            print(super().method())        # B -> C -> A
            print(C.method(self))          # C -> A (прямой вызов)
            print(super(C, self).method()) # A (начать с класса после C)
            
        @classmethod
        def clsmethod(cls):
            print(super().clsmethod())     # B -> C -> A from D
            
        @staticmethod
        def stmethod():
            print(super().stmethod())      # B static -> A static

    # Вызовы вне класса
    d = D()
    d.method()
    D.clsmethod()
    D.stmethod()
    
    print(super(C, C).clsmethod()) # A from C
    print(super(C, D).clsmethod()) # A from D
    print(super(B).stmethod())     # A static

**super и MRO:** всегда следует порядку разрешения методов, что гарантирует корректную работу даже при сложном множественном наследовании.

**Как гарантированно вызвать метод конкретного предка (обход MRO):**

Самый надёжный способ — прямой вызов через имя класса с явной передачей `self`:

    class D(B, C):
        def method(self):
            A.method(self)  # всегда вызовет A.method, независимо от MRO
            B.method(self)  # всегда вызовет B.method
            C.method(self)  # всегда вызовет C.method

Этот способ:
- Не зависит от порядка наследования
- Не сломается при появлении новых классов в MRO
- Понятен при чтении кода

**__init__.py и __all__**  
- `__init__.py` делает директорию пакетом.  
- `__all__ = ['func', 'Class']` определяет, что импортируется при `from package import *`.

**Метаклассы (Metaclasses)**

Метакласс — это класс, который создаёт другие классы. Позволяет перехватывать создание класса и изменять его **до того, как появятся экземпляры**.

**Метакласс vs Наследование vs Декоратор класса**

| Характеристика | Метакласс | Декоратор класса | Наследование |
|---|---|---|---|
| **Синтаксис** | `class MyClass(metaclass=Meta):` | `@decorator class MyClass:` | `class Child(Parent):` |
| **Когда выполняется** | При создании класса | После создания класса | При вызове методов, унаследованных от родителя |
| **Что изменяет** | Процесс создания класса (меняет структуру, атрибуты, методы до появления класса) | Уже созданный класс (меняет структуру, атрибуты, методы после его появления) | Поведение, унаследованное от родителя |
| **Наследуется** | Да | Нет | Да |
| **Типичное применение** | Регистрация всех наследников, валидация структуры до создания | Добавление методов/атрибутов к конкретному классу | Переиспользование логики родителя |

**Параметры методов метакласса:**

- `cls` — сам метакласс (аналог `self` для обычных классов)
- `name` — имя создаваемого класса (строка)
- `bases` — кортеж родительских классов
- `dct` — словарь атрибутов и методов класса

**Основные методы метакласса:**

    class Meta(type):
        def __new__(cls, name, bases, dct):
            # Вызывается ДО создания класса
            # Здесь можно изменить dct (атрибуты класса)
            dct['version'] = 1.0
            dct['new_method'] = lambda self: "добавлено в __new__"
            return super().__new__(cls, name, bases, dct)
        
        def __init__(cls, name, bases, dct):
            # Вызывается ПОСЛЕ создания класса
            # Здесь можно работать с уже созданным классом (cls)
            cls.initialized = True
            super().__init__(name, bases, dct)
        
        def __call__(cls, *args, **kwargs):
            # Вызывается при создании ЭКЗЕМПЛЯРА класса (MyClass())
            # Здесь можно контролировать создание объектов (синглтон, пул и т.д.)
            print(f"Создаётся экземпляр {cls.__name__}")
            instance = super().__call__(*args, **kwargs)
            return instance

    class MyClass(metaclass=Meta):
        pass

    print(MyClass.version)       # 1.0 (добавлено до создания экземпляров)
    print(MyClass.initialized)   # True (после __init__)
    obj = MyClass()              # "Создаётся экземпляр MyClass"
    print(obj.new_method())      # "добавлено в __new__"

**Что происходит по шагам:**

1. Python встречает `class MyClass(metaclass=Meta)`
2. Вызывается `Meta.__new__` — создаётся класс MyClass
3. Вызывается `Meta.__init__` — инициализируется готовый класс
4. При вызове `MyClass()` срабатывает `Meta.__call__`
5. Внутри `__call__` создаётся объект и вызываются `__new__` и `__init__` самого класса

**Зачем нужны метаклассы (реальные примеры):**

    # Единый пример: валидация, автозаполнение, регистрация и синглтон
    plugins = {}

    class UsefulMeta(type):
        def __new__(cls, name, bases, dct):
            # 1. Валидация — требуем обязательный атрибут
            if 'important' not in dct:
                raise TypeError(f"{name} должен определить 'important'")
            
            # 2. Автоматически добавляем атрибуты
            dct['version'] = '1.0'
            
            # 3. Регистрируем класс (кроме базового)
            if name != 'Base':
                plugins[name] = cls  # сам класс в словарь
            
            return super().__new__(cls, name, bases, dct)
        
        def __call__(cls, *args, **kwargs):
            # 4. Синглтон через метакласс — работает для любого класса с этим метаклассом
            if not hasattr(cls, '_instance'):
                cls._instance = super().__call__(*args, **kwargs)
            return cls._instance

    class Base(metaclass=UsefulMeta):
        important = True  # обязательный атрибут

    class User(Base):
        important = 42

    class Product(Base):
        important = "yes"

    # Автоматическая регистрация
    print(plugins)  # {'User': <class '__main__.User'>, 'Product': <class '__main__.Product'>}

    # Автодобавленные атрибуты
    print(User.version)   # '1.0'

    # Синглтон — для каждого класса свой
    a = User()
    b = User()
    p = Product()
    print(a is b)  # True (один класс User)
    print(a is p)  # False (разные классы)

**Важно о синглтоне через метакласс:**  
Один метакласс с `__call__` делает **все** свои классы синглтонами. У каждого класса будет свой единственный экземпляр.

**Может ли у класса быть несколько метаклассов?**

Нет, у класса может быть только один метакласс. При попытке указать несколько будет ошибка:

    class Meta1(type): pass
    class Meta2(type): pass

    # class MyClass(metaclass=Meta1, Meta2): pass  # SyntaxError

Если нужно скомбинировать функциональность, метаклассы можно наследовать:

    class CombinedMeta(Meta1, Meta2): pass
    class MyClass(metaclass=CombinedMeta): pass

**Можно ли без метаклассов?**

Для работы с экземплярами — да (хватит родительского `__new__`).  
Для изменения **самого класса** — метаклассы незаменимы:

- Автоматическое добавление атрибутов класса (`table_name = class.__name__.lower()`)
- Валидация структуры класса (обязательные атрибуты)
- Регистрация классов в реестре (плагины, команды)
- Создание API, похожих на Django ORM или SQLAlchemy


**Дескрипторы (Descriptors)**

Дескриптор — это класс, который перехватывает операции доступа к атрибуту другого класса. Для этого он реализует специальные методы.

**Методы дескриптора:**

- `__get__(self, obj, objtype)` — вызывается при чтении атрибута (`obj.attr`)
  - `self` — сам дескриптор
  - `obj` — экземпляр класса, через который идёт доступ (если через класс — `None`)
  - `objtype` — класс, которому принадлежит дескриптор

- `__set__(self, obj, value)` — вызывается при записи (`obj.attr = value`)
  - `self` — сам дескриптор
  - `obj` — экземпляр класса
  - `value` — присваиваемое значение

- `__delete__(self, obj)` — вызывается при удалении (`del obj.attr`)
  - `self` — сам дескриптор
  - `obj` — экземпляр класса

- `__set_name__(self, owner, name)` — вызывается при создании класса, чтобы дескриптор узнал своё имя
  - `self` — сам дескриптор
  - `owner` — класс, в котором создан дескриптор
  - `name` — имя атрибута (строка)

**Виды дескрипторов:**

| Вид | Какие методы реализует | Примеры |
|-----|------------------------|---------|
| Не-дескриптор | только `__get__` | обычные методы, `@staticmethod`, `@classmethod` |
| Дескриптор данных | `__get__` + `__set__` (может и `__delete__`) | `@property`, валидаторы |

**Пример с пояснениями:**

    class PositiveNumber:
        def __set_name__(self, owner, name):
            self.name = name  # запоминаем имя атрибута ('price')
        
        def __get__(self, obj, objtype=None):
            if obj is None:
                # Доступ через класс (Order.price) — obj.__dict__ не существует
                return self
            # Доступ через экземпляр (order.price)
            return obj.__dict__.get(self.name)
        
        def __set__(self, obj, value):
            if value <= 0:
                raise ValueError("Positive number only")
            obj.__dict__[self.name] = value

    class Order:
        price = PositiveNumber()  # price — дескриптор

    order = Order()
    order.price = 100    # вызывает __set__
    print(order.price)   # вызывает __get__(order, Order)
    print(Order.price)   # вызывает __get__(None, Order) → возвращает дескриптор

**Как это связано с @property, @classmethod, @staticmethod?**

Запись с `@` — это синтаксический сахар. На самом деле там создаётся дескриптор.

    class MyClass:
        @property
        def x(self): ...

    # Превращается в:
    class MyClass:
        def x(self): ...
        x = property(x)  # property — это дескриптор

    class MyClass:
        @classmethod
        def create(cls): ...

    # Превращается в:
    class MyClass:
        def create(cls): ...
        create = classmethod(create)  # classmethod — тоже дескриптор

    class MyClass:
        @staticmethod
        def helper(): ...

    # Превращается в:
    class MyClass:
        def helper(): ...
        helper = staticmethod(helper)  # staticmethod — дескриптор

То есть `property`, `classmethod`, `staticmethod` — это **готовые классы-дескрипторы** из стандартной библиотеки.

**Главное отличие @property от остальных:**

- `property` — дескриптор данных (есть `__set__`), поэтому перехватывает и запись
- `classmethod` и `staticmethod` — не-дескрипторы (только `__get__`), поэтому влияют только на чтение


### Управление ресурсами (контекстные менеджеры)

**Контекстные менеджеры**

Используются для управления ресурсами (файлы, соединения, блокировки). Гарантируют освобождение ресурса даже при ошибке.

**Через класс с __enter__ и __exit__:**

    class FileOpener:
        def __init__(self, filename, mode):
            self.filename = filename
            self.mode = mode
        
        def __enter__(self):
            self.file = open(self.filename, self.mode)
            return self.file  # попадёт в переменную after as
        
        def __exit__(self, exc_type, exc_val, exc_tb):
            # exc_type — класс исключения (None, если ошибки не было)
            # exc_val — объект исключения
            # exc_tb — traceback
            self.file.close()
            # return True — подавить исключение, False (или None) — пробросить дальше

**Через @contextmanager из contextlib (синтаксический сахар):**

    from contextlib import contextmanager

    @contextmanager
    def file_opener(filename, mode):
        file = open(filename, mode)  # __enter__
        try:
            yield file  # выполняется блок with
        finally:
            file.close()  # __exit__ — сработает даже при ошибке

**Использование (одинаково для обоих способов):**

    with file_opener('test.txt', 'w') as f:
        f.write('Hello')

**Где ещё встречается:**

    with open('file.txt') as f:          # файлы
        data = f.read()


## 2. Алгоритмы и структуры данных

### Сложность алгоритмов (Big O, Θ, Ω)

Оценка того, как изменяется время выполнения (или объём используемой памяти) при изменении n — размера входных данных. Измеряется в количестве базовых операций:
- **Для поиска:** сравнение элементов
- **Для сортировки:** сравнение и перестановка элементов

**Обозначения:**
- **O (О большое)** — верхняя граница, худший случай
- **Ω (Омега)** — нижняя граница, лучший случай
- **Θ (Тета)** — точная граница для среднего случая (если лучший и худший совпадают — просто Θ, если нет — указывается отдельно)

| Обозначение | Название | Примеры |
|---|---|---|
| O(1) | Константная | Доступ по индексу в списке, поиск в хеш-таблице |
| O(log n) | Логарифмическая | Бинарный поиск в отсортированном массиве |
| O(n) | Линейная | Поиск элемента в списке, проход по всем элементам |
| O(n log n) | Линейно-логарифмическая | Сортировка слиянием, быстрая сортировка (в среднем), Timsort |
| O(n²) | Квадратичная | Вложенные циклы, пузырьковая сортировка, быстрая сортировка (в худшем) |

### Структуры данных

#### Список (list)
- Динамический массив. Элементы хранятся последовательно в памяти. По индексу можно вычислить адрес элемента одной операцией: `адрес = начало + индекс × размер_элемента`.
- Доступ по индексу: O(1) — всегда
- Поиск элемента по значению: O(n) — всегда
- Вставка/удаление в конце: амортизированно O(1) (изредка случается расширение массива, но в среднем по последовательности операций — константа)
- Вставка/удаление в начале/середине: O(n) — всегда

#### Кортеж (tuple)
- Неизменяемый аналог списка
- Доступ по индексу: O(1) — всегда
- Поиск элемента по значению: O(n) — всегда
- Вставка/удаление: ❌ не поддерживается

#### Множество (set)
- Хеш-таблица (только ключи). По значению вычисляется хеш, который преобразуется в индекс массива, а затем в адрес элемента: `адрес = начало + (hash(value) % size) × размер_ячейки`.
- Проверка вхождения: в среднем Θ(1), в худшем O(n) (коллизии)
- Добавление: в среднем Θ(1), в худшем O(n)
- Удаление: в среднем Θ(1), в худшем O(n)

#### Frozenset
- Неизменяемое множество
- Проверка вхождения: в среднем Θ(1), в худшем O(n)
- Добавление/удаление: ❌ не поддерживается

#### Словарь (dict)
- Хеш-таблица (ключи → значения). По ключу вычисляется хеш, который преобразуется в индекс массива, а затем в адрес элемента: `адрес = начало + (hash(key) % size) × размер_ячейки`.
- Поиск по ключу: в среднем Θ(1), в худшем O(n) (коллизии)
- Вставка по ключу: в среднем Θ(1), в худшем O(n)
- Удаление по ключу: в среднем Θ(1), в худшем O(n)
- Поиск по значению: O(n) — всегда

#### Стек (stack)
- Принцип LIFO (последним пришёл — первым ушёл)
- В Python: list (append/pop) или collections.deque
- Добавление (push): O(1)
- Удаление (pop): O(1)

#### Очередь (queue)
- Принцип FIFO (первым пришёл — первым ушёл)
- В Python: collections.deque (append/popleft) или queue.Queue
- Добавление (enqueue): O(1)
- Удаление (dequeue): O(1)

#### Графы
- **Определение:** набор вершин (узлов) и рёбер (связей) между ними.
- Обозначения:
  - **V** (vertices) — количество вершин
  - **E** (edges) — количество рёбер
- Могут быть:
  - **ориентированными** (рёбра имеют направление) или **неориентированными** (связь двусторонняя)
  - **взвешенными** (у каждого ребра есть число — стоимость, расстояние) или **невзвешенными** (все рёбра равнозначны)
- Хранение: списки смежности (dict: вершина → список соседей) или матрица смежности

#### Деревья
- **Определение:** связный граф без циклов.
- **Бинарное дерево поиска:** у каждого узла не больше двух потомков, при этом левый потомок меньше родителя, правый — больше.
- Поиск/вставка/удаление: в среднем O(log n), в худшем O(n) (если дерево выродилось в список).
- Обходы: прямой (pre-order), симметричный (in-order), обратный (post-order), в ширину (level-order).

#### Куча (heap)
- **Определение:** двоичное дерево, где каждый родитель меньше (min-heap) или больше (max-heap) своих потомков.
- В Python: модуль `heapq` (только min-heap).
- Получение минимума: O(1)
- Добавление (heappush): O(log n)
- Удаление минимума (heappop): O(log n)


    import heapq
    
    heap = []
    heapq.heappush(heap, 5)
    heapq.heappush(heap, 2)
    heapq.heappush(heap, 7)
    
    smallest = heapq.heappop(heap)  # 2


- Применение: очередь с приоритетом, алгоритм Дейкстры, задачи на поиск k-го элемента.

### Алгоритмы на графах

#### Обходы
- **Поиск в глубину (DFS)** — идёт вглубь, использует стек (или рекурсию). Сложность: O(V + E).
- **Поиск в ширину (BFS)** — идёт по уровням, использует очередь. Находит кратчайший путь в невзвешенном графе. Сложность: O(V + E).

#### Поиск кратчайших путей
- **Алгоритм Дейкстры** — кратчайшие пути от одной вершины до всех остальных во взвешенном графе без отрицательных рёбер. Сложность: O(E log V) с приоритетной очередью.
- **Алгоритм Беллмана-Форда** — работает с отрицательными рёбрами, обнаруживает отрицательные циклы. Сложность: O(V·E).
- **Алгоритм Флойда-Уоршелла** — кратчайшие пути между всеми парами вершин. Сложность: O(V³).

#### Другие алгоритмы на графах
- **Топологическая сортировка** — упорядочивает вершины ориентированного графа без циклов так, что все рёбра идут слева направо. Использует DFS или алгоритм Кана.
- **Алгоритм Прима** — минимальное остовное дерево (MST) для связного взвешенного графа. Жадный алгоритм.
- **Алгоритм Краскала** — тоже для MST, использует систему непересекающихся множеств (DSU).

### Алгоритмы поиска (в массивах)

- **Линейный поиск** — O(n). Перебор всех элементов до нахождения нужного.
- **Бинарный поиск** — O(log n). Только для отсортированных данных. Делит массив пополам на каждом шаге.

### Алгоритмы сортировки

- **Пузырьковая сортировка** — O(n²). Попарные сравнения и обмены соседних элементов. Только для понимания.
- **Быстрая сортировка (quick sort)** — в среднем Θ(n log n), в худшем O(n²). Выбирает опорный элемент, делит на меньшие/большие, рекурсивно сортирует части.
- **Сортировка слиянием (merge sort)** — Θ(n log n) всегда. Делит массив пополам, рекурсивно сортирует части, сливает отсортированные части.
- **Timsort** — встроенная сортировка Python. Гибрид сортировки слиянием (merge sort) и сортировки вставками (insertion sort). Θ(n log n).

### Динамическое программирование

Идея: разбить задачу на подзадачи, сохранять результаты, чтобы не пересчитывать заново.

- **Мемоизация (сверху вниз)** — рекурсия + кэш (сохраняем результаты уже посчитанных подзадач). Пример: числа Фибоначчи.

      def fib(n, memo={}):
          if n in memo:
              return memo[n]
          if n <= 1:
              return n
          memo[n] = fib(n-1, memo) + fib(n-2, memo)
          return memo[n]

- **Табуляция (снизу вверх)** — заполняем таблицу итеративно, начиная с базовых случаев. Пример: числа Фибоначчи.

      def fib(n):
          if n <= 1:
              return n
          dp = [0] * (n + 1)
          dp[1] = 1
          for i in range(2, n + 1):
              dp[i] = dp[i-1] + dp[i-2]
          return dp[n]

**Задача о рюкзаке (0/1 knapsack):** есть предметы с весом и стоимостью, нужно набрать рюкзак вместимостью W так, чтобы стоимость была максимальной. Каждый предмет можно взять 0 или 1 раз.

      def knapsack(weights, values, W):
          n = len(weights)
          dp = [[0] * (W + 1) for _ in range(n + 1)]
          
          for i in range(1, n + 1):
              for w in range(1, W + 1):
                  if weights[i-1] <= w:
                      dp[i][w] = max(
                          values[i-1] + dp[i-1][w - weights[i-1]],
                          dp[i-1][w]
                      )
                  else:
                      dp[i][w] = dp[i-1][w]
          return dp[n][W]

Сложность: O(n·W) по времени и памяти (можно оптимизировать до O(W) по памяти).
## 3. Коварные вопросы (что выведет код?)

### Пример 1: Аргументы по умолчанию

    def add_element(value, target=[]):
        target.append(value)
        return target

    print(add_element(1))       # [1]
    print(add_element(2))       # [1, 2]  (а не [2]!)
    print(add_element(3, []))   # [3]
    print(add_element(4))       # [1, 2, 4]

**Почему:** Аргументы по умолчанию создаются один раз при определении функции, а не при каждом вызове. Все вызовы, не передающие свой список, используют один и тот же объект.

**Важные нюансы:**

- Каждый параметр со значением по умолчанию получает **свой собственный** постоянный объект:
  
      def two_defaults(x, lst1=[], lst2=[]):
          lst1.append(x)
          lst2.append(x*10)
          return lst1, lst2
      
      print(two_defaults(1))  # ([1], [10])
      print(two_defaults(2))  # ([1,2], [10,20]) — два независимых общих списка

- У разных функций — **разные** объекты по умолчанию, даже если значения выглядят одинаково:

      def f1(x, lst=[]): ...
      def f2(x, lst=[]): ...  # это другой список, не общий с f1

- В Pydantic нельзя использовать изменяемые объекты в качестве значений по умолчанию — нужно использовать `Field(default_factory=list)`, чтобы каждый экземпляр модели получал свой список.

**Решение:** всегда использовать `None` для изменяемых значений по умолчанию:

    def add_element(value, target=None):
        if target is None:
            target = []
        target.append(value)
        return target

### Пример 2: Замыкания в цикле

    funcs = []
    for i in range(3):
        funcs.append(lambda: i)

    for f in funcs:
        print(f(), end=' ')  # 2 2 2 (а не 0 1 2)

**Почему:** Все лямбды используют последнее значение i из внешней области видимости после завершения цикла. Решение: `lambda i=i: i` или `lambda x=i: x`.

### Пример 3: Цепочка исключений

    try:
        print(undefined_var)
    except NameError:
        print("Caught")
        raise ValueError("New error")

**Что будет:** В Python 3 при повторном поднятии исключения информация об исходной ошибке (NameError) сохраняется в цепочке исключений. В Python 2 исходная ошибка теряется, виден только ValueError.

### Пример 4: Множественное наследование и super()

    class A:
        def __init__(self):
            print("A", end='')
            super().__init__()

    class B:
        def __init__(self):
            print("B", end='')
            super().__init__()

    class C(A, B):
        def __init__(self):
            print("C", end='')
            super().__init__()

    C()  # CAB (порядок согласно MRO)

**Почему:** MRO для C: (C, A, B, object). super() следует этому порядку.

### Пример 5: Изменяемые объекты в кортежах

    t = (1, [2, 3])
    t[1].append(4)
    print(t)  # (1, [2, 3, 4])

**Почему:** Кортеж хранит не сами объекты, а ссылки на них. Сам кортеж неизменяем — нельзя заменить ссылку (например, `t[1] = [5,6]` вызовет ошибку). Но если ссылка ведёт на изменяемый объект (список), этот объект можно менять.

### Пример 6: Замыкание и позднее связывание

    x = 10
    def outer():
        x = 20
        def inner():
            print(x)
        return inner

    func = outer()
    func()  # 20 (лексическая область видимости)

**Почему:** inner запоминает значение x из внешней функции outer, а не глобальное x.

### Пример 7: Оператор is и малые целые числа

    a = 256
    b = 256
    print(a is b)  # True

    c = 257
    d = 257
    print(c is d)  # False

**Почему:** Python заранее создаёт в памяти один объект для каждого числа от -5 до 256 и переиспользует его при каждом обращении. Для чисел вне этого диапазона каждый раз создаётся новый объект.

### Пример 8: Генераторы и исключения

    def gen():
        try:
            yield 1
            yield 2
        except GeneratorExit:
            print("Generator closed")

    g = gen()
    print(next(g))  # 1
    g.close()       # Generator closed
    print(next(g))  # StopIteration

**Почему:** При закрытии генератора выбрасывается GeneratorExit, который можно перехватить.

### Пример 9: Срезы за пределами

    lst = [1, 2, 3]
    print(lst[10:20])  # [] (пустой список)
    print(lst[10])     # IndexError
    print(lst[-1])     # 3 (последний элемент)
    print(lst[-5])     # IndexError (выход за границы слева)

**Почему:** Срезы терпимы к выходу за границы — если указан диапазон вне списка, возвращается пустой список. При доступе по одному индексу Python жёстко проверяет границы: индекс должен быть в диапазоне от `-len(lst)` до `len(lst)-1` включительно. `-1` даёт последний элемент, `-2` — предпоследний и так далее до `-len(lst)`. За этими пределами — IndexError.

### Пример 10: Удаление элемента во время итерации

    lst = [1, 2, 3, 4]
    for i, x in enumerate(lst):
        if x % 2 == 0:
            del lst[i]
    print(lst)  # [1, 3] (может работать, но опасно!)

**Почему:** Изменение списка во время итерации приводит к пропуску элементов или IndexError. Правильно: создавать новый список или итерировать по копии.

### Пример 11: Логические операторы

    print(False and 1/0)  # False (1/0 не вычисляется)
    print(True or 1/0)    # True (1/0 не вычисляется)
    print([] or [1,2])    # [1,2] (возвращает первый истинный)
    print([1,2] and [])   # [] (возвращает последний операнд)
    print([] and [1,2])   # [] (возвращает первый ложный)

**Почему:** `and` и `or` вычисляются лениво (short-circuit):
- `and` возвращает первый ложный операнд, если такой есть, иначе последний
- `or` возвращает первый истинный операнд, если такой есть, иначе последний

### Пример 12: Имена функций и переменных

    def func():
        pass

    func = 42
    print(func)  # 42 (func теперь число)

**Почему:** В Python функции — объекты первого класса. Имя функции — это просто переменная, которая хранит ссылку на объект-функцию. Как и любую другую переменную, её можно переназначить на любой объект (число, строку и т.д.). При этом сам объект-функция продолжает существовать в памяти, но обратиться к нему через старое имя уже нельзя.

**Объекты первого класса (first-class objects)** — объекты, которые можно:
- передавать в функции как аргументы
- возвращать из функций как результат
- присваивать переменным
- хранить в структурах данных (списках, словарях)

В Python **всё** является объектами первого класса: числа, строки, функции, классы, модули.

**Объекты второго класса** — имеют ограничения (нельзя передать как аргумент). Пример: функции в C (можно передать указатель, но это не сам объект).

**Объекты третьего класса** — ещё больше ограничений (например, метки goto в C).

## 4. Асинхронность, потоки и процессы

### Процессы и потоки

**Процесс** — экземпляр программы с изолированным адресным пространством, собственными ресурсами и как минимум одним потоком. Процессы не разделяют память.

**Поток** — единица выполнения внутри процесса (поток ОС). Все потоки одного процесса разделяют его память и конкурируют за GIL.

Любой процесс имеет главный поток. Из любого потока процесса можно создавать другие потоки (в том же процессе) и новые процессы.

Когда создаётся новый процесс, у него появляется собственное адресное пространство, свой интерпретатор и **свой собственный GIL**. Потоки внутри дочернего процесса конкурируют за его собственный GIL, который не зависит от родительского.

### Конкурентность, Асинхронность, Параллелизм

**Конкурентность** — умение управлять несколькими задачами так, будто они выполняются одновременно. Задачи переключаются, пока одна ждёт (например, ответ от сервера). Пример: повар поставил варить суп и, пока суп варится, чистит картошку — он конкурентно управляет несколькими процессами, хотя физически делает только одно дело в каждый момент времени.

**Асинхронность** — частный случай конкурентности. Задачи выполняются в одном потоке, сами отдают управление в точках `await`. Использует событийный цикл, нет переключений потоков ОС. Подходит для I/O-bound задач с тысячами соединений.

**Параллелизм** — физическое выполнение нескольких задач в одно и то же время на разных ядрах процессора. Пример: два повара на разных кухнях: один варит суп, другой жарит котлеты — одновременно.

### Способы организации кода

**Синхронный код**  
Выполняется последовательно. При ожидании I/O поток блокируется.  
*Пример:* обычная функция, читающая файл.

**Асинхронный код (asyncio)**  
В одном потоке ОС выполняются корутины. Для запуска корутины в фоне создаётся задача (`asyncio.create_task`). Переключение между задачами — только в точках `await` (кооперативная многозадачность). GIL не мешает, так как поток один.  
*Подходит для:* высоконагруженных I/O-bound систем (тысячи веб-сокетов, долгих запросов).

**Потоки (threading)**  
Потоки ОС выполняют синхронный код. Из-за GIL одновременно выполняется только один поток с Python-кодом. Переключение — на уровне ОС: при I/O (GIL освобождается) или по истечении кванта времени (даже в середине Python-кода). Требует синхронизации через блокировки.  
*Подходит для:* I/O-bound задач, когда есть готовый синхронный код.

**Многопроцессорность (multiprocessing)**  
Несколько процессов, каждый со своим GIL. Позволяет задействовать все ядра для CPU-bound задач. Обмен данными (IPC) сложнее, больше накладных расходов.  
*Подходит для:* CPU-bound задач.

### Сравнение asyncio, threading и multiprocessing

| Аспект | asyncio | threading | multiprocessing |
| :--- | :--- | :--- | :--- |
| **Параллелизм CPU** | Нет (один поток) | Нет (GIL) | Да (отдельные процессы) |
| **I/O задачи** | Очень эффективен (GIL освобождается) | Эффективен (GIL освобождается) | Менее эффективен из-за создания процессов и IPC (GIL освобождается) |
| **I/O одновременно** | Тысячи | Сотни | Десятки |
| **Стиль кода** | Асинхронный (async/await) | Синхронный | Синхронный |
| **Память** | Разделяемая | Разделяемая | Изолированная |
| **Сложность реализации** | Высокая (event loop, async/await) | Низкая (блокировки) | Средняя (обмен данными между процессами) |


### Синхронизация и передача данных

**Asyncio (asyncio)**
- Все задачи в одном потоке, гонок данных нет
- `asyncio.Lock` — блокировка для задач. Задача, которая пытается захватить уже занятую блокировку, приостанавливается до её освобождения.
- `ContextVar` — свои данные для каждой задачи
- `call_soon_threadsafe` — отправляет функцию в событийный цикл для выполнения в том потоке, в котором этот цикл был запущен

**Потоки (threading)**
- Общая память (быстро, нужны блокировки)
- `threading.Lock` — блокировка для потоков. Поток, который пытается захватить уже занятую блокировку, блокируется до её освобождения.
- `threading.local()` — свои данные для каждого потока

**Процессы (multiprocessing)**
- Память изолирована, обмен через IPC (Inter-Process Communication — механизмы обмена данными между процессами):
  - `Queue`, `Pipe` — передача данных через очереди или каналы (как сообщения)
  - `Value`, `Array` — общая память, доступная нескольким процессам (требует синхронизации, чтобы избежать гонок)
  - `Manager` — создаёт общие объекты (списки, словари), доступные из разных процессов (удобно, но медленнее из-за дополнительной обёртки)

### Потокобезопасность

**Потокобезопасность** — свойство объекта или функции, которым можно пользоваться из нескольких потоков одновременно, не опасаясь повреждения данных или неопределённого поведения.

**Гонка данных (race condition)** — ситуация, когда несколько потоков одновременно обращаются к общим данным, и хотя бы один из них пишет. Результат зависит от порядка выполнения и непредсказуем.

    counter = 0

    def increment():
        global counter
        for _ in range(100000):
            current = counter  # чтение
            counter = current + 1  # запись

    # Два потока одновременно выполняют increment
    # Ожидалось: 200000, реально: меньше (например, 142387)

**Атомарные операции** — операции, выполняемые целиком, без прерывания другим потоком. В CPython благодаря GIL некоторые операции атомарны:
- Простое присваивание: `x = 1`
- Чтение/запись атрибута: `obj.attr = value`
- Вызов функций (но не их тело)

Однако составные операции, такие как `x += 1` (чтение + запись), неатомарны и могут вызывать гонки данных.

**Как обеспечить потокобезопасность:**

- **Lock** — блокировка, которую может захватить только один поток

      lock = threading.Lock()
      with lock:
          counter += 1  # атомарно внутри блока

- **threading.local()** — у каждого потока своя копия данных (нет общего состояния)

- **queue.Queue** — потокобезопасная очередь для обмена данными

- **В asyncio** проблемы гонок данных практически нет (всё в одном потоке), но если нужно взаимодействовать с threading — используют **call_soon_threadsafe**

**Пример: потокобезопасный Event для asyncio**

    import asyncio

    class Event_ts(asyncio.Event):
        def __init__(self, *args, **kwargs):
            super().__init__(*args, **kwargs)
            self._loop = asyncio.get_running_loop()

        def set(self):
            self._loop.call_soon_threadsafe(super().set)

        def clear(self):
            self._loop.call_soon_threadsafe(super().clear)

Обычный `asyncio.Event` можно использовать только внутри потока с событийным циклом. Если вызвать `event.set()` из другого потока (например, из `threading`), это нарушит потокобезопасность. `Event_ts` решает проблему: вызовы `set`/`clear` планируются в правильном потоке через `call_soon_threadsafe`.

## 5. Базы данных (SQL)

### Нормальные формы

---

**1НФ (первая нормальная форма)**  
Требование: в каждой ячейке одно значение, все строки уникальны.

**Исходная таблица:**

| id | Имя | Курсы |
|----|-----|-------|
| 1  | Иванов | Математика, Физика |
| 2  | Петров | История |

*Нарушение:* в ячейке "Курсы" перечислено несколько курсов.  
*Решение:* разбить строки.

**Результат (1НФ).** Составной ключ: (id_студента, id_курса):

| id_студента | id_курса | Имя студента | Название курса |
|-------------|----------|--------------|----------------|
| 1           | 101      | Иванов       | Математика     |
| 1           | 102      | Иванов       | Физика         |
| 2           | 103      | Петров       | История        |

---

**2НФ (вторая нормальная форма)**  
Требование: таблица в 1НФ, и все неключевые атрибуты зависят от всего первичного ключа (устранение частичной зависимости).

Ключ составной: (id_студента, id_курса).

*Нарушение:*  
- Имя студента зависит только от id_студента  
- Название курса зависит только от id_курса  

*Решение:* разбить на три таблицы.

**Студенты:**

| id | Имя |
|----|-----|
| 1  | Иванов |
| 2  | Петров |

**Курсы:**

| id | Название |
|----|----------|
| 101| Математика |
| 102| Физика |
| 103| История |

**Записи:**

| id_студента | id_курса |
|-------------|----------|
| 1           | 101      |
| 1           | 102      |
| 2           | 103      |

---

**3НФ (третья нормальная форма)**  
Требование: таблица в 2НФ и нет транзитивных зависимостей (неключевые атрибуты не зависят от других неключевых).

**Исходная таблица** (ключ: id):

| id | Имя | код_группы | Название_группы |
|----|-----|------------|-----------------|
| 1  | Иванов | 101 | ИТ-31 |
| 2  | Петров | 102 | ИТ-32 |

*Нарушение:* Название_группы зависит от код_группы, а не от id студента.  
*Решение:* вынести группы в отдельную таблицу.

**Студенты:**

| id | Имя | код_группы |
|----|-----|------------|
| 1  | Иванов | 101 |
| 2  | Петров | 102 |

**Группы:**

| код_группы | Название |
|------------|----------|
| 101        | ИТ-31    |
| 102        | ИТ-32    |

---

**НФБК (нормальная форма Бойса-Кодда)**  
Требование: таблица в 3НФ, и ключевые атрибуты не зависят от неключевых.

**Исходная таблица** (ключ: teacher_id, course_id):

| teacher_id | course_id | room_id |
|------------|-----------|---------|
| 1          | 101       | 201     |
| 1          | 102       | 201     |
| 2          | 103       | 202     |

*Правило:* в одной аудитории может вести только один преподаватель (при этом у преподавателя может быть несколько своих аудиторий).  
*Зависимость:* `room_id` → `teacher_id`.  
Ключевой атрибут `teacher_id` зависит от неключевого атрибута `room_id`. Это нарушение НФБК.

*Решение:* разбить на две таблицы.

**CoursesRooms:**

| course_id | room_id |
|-----------|---------|
| 101       | 201     |
| 102       | 201     |
| 103       | 202     |

**RoomsTeachers:**

| room_id | teacher_id |
|---------|------------|
| 201     | 1          |
| 202     | 2          |

---

**4НФ (четвертая нормальная форма)**  
Требование: таблица в НФБК и отсутствие многозначных зависимостей. Многозначная зависимость возникает, когда один атрибут определяет несколько независимых наборов значений, которые не связаны между собой (например, преподаватель может вести несколько предметов и говорить на нескольких языках, и эти наборы независимы).

**Исходная таблица** (ключ: teacher_id, subject_id, language_id):

| teacher_id | subject_id | language_id |
|------------|------------|-------------|
| 1          | 101        | 1           |
| 1          | 101        | 2           |
| 1          | 102        | 1           |
| 1          | 102        | 2           |

*Нарушение:* преподаватель может вести несколько предметов и говорить на нескольких языках — эти наборы независимы, что порождает избыточность.  
*Решение:* разбить на две таблицы.

**TeacherSubjects:**

| teacher_id | subject_id |
|------------|------------|
| 1          | 101        |
| 1          | 102        |

**TeacherLanguages:**

| teacher_id | language_id |
|------------|-------------|
| 1          | 1           |
| 1          | 2           |

---

**5НФ (пятая нормальная форма)**  
Требование: таблица в 4НФ и отсутствие лишних зависимостей соединения. Лишние зависимости соединения имеются, если из таблицы нельзя выделить две таблицы, которые полностью восстанавливают исходные данные, но можно выделить три (или более) для этого.

**Исходная таблица** (не в 5НФ):

| student_id | course_id | teacher_id |
|------------|-----------|------------|
| 1          | 101       | 1          |
| 1          | 102       | 1          |
| 2          | 101       | 1          |
| 2          | 102       | 1          |

*Правило:* студент должен быть записан к преподавателю на курс.

*Нарушение:* из таблицы нельзя выделить две таблицы, которые полностью восстанавливают исходные данные. Например, если выделить таблицы StudentCourse и CourseTeacher:

**StudentCourse:**

| student_id | course_id |
|------------|-----------|
| 1          | 101       |
| 1          | 102       |
| 2          | 101       |
| 2          | 102       |

**CourseTeacher:**

| course_id | teacher_id |
|-----------|------------|
| 101       | 1          |
| 102       | 2          |

При их соединении получатся лишние записи:

| student_id | course_id | teacher_id |
|------------|-----------|------------|
| 1          | 101       | 1          |
| 1          | 102       | 2          |
| 2          | 101       | 1          |
| 2          | 102       | 2          |

Здесь появилась запись (student_id=1, teacher_id=2), которой не было в исходной таблице. При этом разбиение на три таблицы даёт корректное восстановление.

*Решение:* разбить на три таблицы.

**StudentCourse** (какие курсы выбрал студент):

| student_id | course_id |
|------------|-----------|
| 1          | 101       |
| 1          | 102       |
| 2          | 101       |
| 2          | 102       |

**CourseTeacher** (какой преподаватель ведёт курс):

| course_id | teacher_id |
|-----------|------------|
| 101       | 1          |
| 102       | 2          |

**StudentTeacher** (какой преподаватель закреплён за студентом):

| student_id | teacher_id |
|------------|------------|
| 1          | 1          |
| 2          | 2          |

Теперь таблицы находятся в 5НФ. Исходная таблица восстанавливается соединением всех трёх:

    SELECT * FROM StudentCourse
    JOIN CourseTeacher ON StudentCourse.course_id = CourseTeacher.course_id
    JOIN StudentTeacher ON StudentCourse.student_id = StudentTeacher.student_id
        AND CourseTeacher.teacher_id = StudentTeacher.teacher_id

На практике 5НФ нужна крайне редко.

**Нормализация**
- Плюсы: целостность, согласованность, отсутствие дублирования, гибкость.
- Минусы: сложные запросы (много JOIN), замедление чтения.

**Денормализация** — намеренное отступление от НФ для ускорения чтения (часто в хранилищах данных).

### Индексы

**Назначение:** ускорение поиска. Плата за это — замедление вставки/обновления.

**Виды индексов:**

- **PRIMARY KEY** — первичный ключ. Автоматически создаёт уникальный индекс по полю (или нескольким).
- **UNIQUE** — уникальный индекс. Гарантирует, что значения в столбце (или комбинации) не повторяются.
- **Обычный (неуникальный) индекс** — ускоряет поиск по часто используемым полям в `WHERE`, `JOIN`, `ORDER BY`.
- **Составной (композитный) индекс** — по нескольким полям.  
  *Важен порядок столбцов:* сначала поля с высокой селективностью (много уникальных значений) или те, по которым идёт точное сравнение (`=`), потом — диапазоны (`>`, `<`). Если часто выполняется сортировка по полю, его стоит добавить в индекс (особенно если оно уже используется в фильтрации).
- **Полнотекстовый индекс (FULLTEXT, GIN)** — для поиска по тексту. Позволяет искать слова, словосочетания, учитывать морфологию (в PostgreSQL — через `tsvector` и `tsquery`), а не просто подстроку.
- **Пространственный индекс (SPATIAL)** — для геоданных (точки, линии, полигоны). В отличие от обычных индексов, поддерживает операции типа "найти все точки внутри полигона" или "ближайшие объекты". Пример: `CREATE INDEX ON cities USING GIST (location)` (в PostgreSQL).
- **Частичный (условный) индекс (PostgreSQL)** — только для строк, удовлетворяющих условию. Экономит место и ускоряет выборку по подмножеству данных.  
  `CREATE INDEX idx_active ON users (last_login) WHERE is_active = true`
- **Функциональный индекс** — индекс по результату функции или выражения.  
  `CREATE INDEX idx_lower_email ON users (LOWER(email))`

**Как анализировать запросы:**

Команда `EXPLAIN` показывает план выполнения запроса: какие индексы используются, в каком порядке соединяются таблицы, сколько строк обрабатывается.

    EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';

Результат может содержать:
- `Seq Scan` — полное сканирование таблицы (плохо, если таблица большая)
- `Index Scan` — сканирование по индексу (хорошо)
- `Bitmap Index Scan` — комбинирование нескольких индексов
- `cost` — примерная стоимость операции (чем меньше, тем быстрее)

Если запрос медленный, `EXPLAIN` помогает понять, почему индекс не используется (например, из-за неявного преобразования типов или неправильного порядка полей в составном индексе).

**Важно:** не переиндексировать — каждый индекс замедляет `INSERT`, `UPDATE`, `DELETE`. Добавлять индексы только под реальные запросы.

### Как ускорить работу с БД (помимо индексов)

- **Денормализация** — добавляем избыточные данные в таблицы, чтобы читать быстрее (меньше JOIN). Плата — усложнение вставки и обновления.
- **Партиционирование** — большая таблица физически разбивается на части по диапазону (например, по датам). Запрос читает только нужную часть.
- **Шардирование** — данные распределяются по разным серверам (горизонтальное масштабирование). Каждый сервер отвечает за свою часть данных.
- **Кэширование** — результаты частых запросов сохраняются в Redis/Memcached, чтобы не ходить в БД.
- **Пул соединений (connection pooling)** — соединения с БД не создаются заново для каждого запроса, а переиспользуются.
- **Анализ запросов через EXPLAIN** — см. выше.
- **Избегание SELECT \*** — запрашивайте только нужные колонки, особенно если в таблице много данных или есть поля с большими объёмами (BLOB, TEXT). Меньше данных — быстрее передача и обработка.

### Типы SQL-команд

**DDL (Data Definition Language)** — команды определения схемы данных:
- `CREATE TABLE`
- `ALTER TABLE`
- `DROP TABLE`
- `TRUNCATE TABLE` — быстрое удаление всех строк, сбрасывает автоинкремент, нельзя откатить (в некоторых СУБД)

**DML (Data Manipulation Language)** — команды манипуляции данными:
- `SELECT`
- `INSERT`
- `UPDATE`
- `DELETE` — построчное удаление (можно с WHERE), медленнее, можно откатить, не сбрасывает автоинкремент

**DCL (Data Control Language)** — команды управления доступом:
- `GRANT`
- `REVOKE`

**TCL (Transaction Control Language)** — команды управления транзакциями:
- `COMMIT`
- `ROLLBACK`
- `SAVEPOINT`

### JOIN

**JOIN (все виды на примере `Employees` (emp) и `Departments` (dep) с `emp.dep_id = dep.id`)**  
- **INNER JOIN** (или просто JOIN): только сотрудники с отделом (и только отделы с сотрудниками).  
- **LEFT JOIN** (LEFT OUTER JOIN): все сотрудники. Если отдела нет — поля dep будут NULL.  
- **RIGHT JOIN** (RIGHT OUTER JOIN): все отделы. Если сотрудников нет — поля emp будут NULL.  
- **FULL JOIN** (FULL OUTER JOIN): все сотрудники и все отделы. Если нет связи — недостающие поля NULL.  
  *При нескольких строках в одной таблице на одну строку другой — строки перемножаются.*  
- **CROSS JOIN:** декартово произведение (каждый с каждым), без условия.  
- **SELF JOIN:** объединение таблицы с самой собой (иерархии).

*Обязательность написания:* `INNER` можно опустить. `OUTER` обычно опускают (`LEFT JOIN` = `LEFT OUTER JOIN`).

**Как JOIN выполняется внутри (упрощённо)**

Когда вы пишете `JOIN`, база данных выбирает один из трёх способов соединить таблицы:

**Nested Loop Join (вложенные циклы)**  
Берётся каждая строка из первой таблицы и для неё ищется совпадение во второй таблице. Если во второй таблице есть индекс — поиск быстрый, если нет — полное сканирование. Хорошо работает, когда одна из таблиц маленькая.

**Hash Join (соединение через хэш)**  
Сначала строится хэш-таблица из меньшей таблицы (ключ — поле соединения, значение — строка). Затем проходим по большей таблице, для каждой строки вычисляем хэш и смотрим, есть ли совпадение в хэш-таблице. Эффективно для больших неиндексированных таблиц.

**Merge Join (соединение слиянием)**  
Обе таблицы сортируются по ключу соединения (если ещё не отсортированы), затем проходятся одновременно как два списка. Хорошо работает, когда таблицы уже отсортированы (например, по индексу).

Планировщик выбирает способ автоматически, оценивая размер таблиц, наличие индексов и статистику.

### Оконные функции (Window Functions)

Выполняют вычисления для набора строк, связанных с текущей строкой, не сворачивая их (в отличие от GROUP BY). Синтаксис: `func() OVER (PARTITION BY колонка_для_группировки ORDER BY колонка_для_сортировки)` — PARTITION BY и ORDER BY необязательны, можно использовать по отдельности или опустить.

**Виды оконных функций:**

- **Ранжирующие (Ranking functions):**  
  `ROW_NUMBER()` — порядковый номер строки (1,2,3...)  
  `RANK()` — номер с пропусками после одинаковых значений (1,1,3...)  
  `DENSE_RANK()` — номер без пропусков (1,1,2...)  
  `NTILE(n)` — разбиение на n групп примерно равного размера

- **Агрегатные (Aggregate functions):**  
  `SUM()`, `AVG()`, `COUNT()`, `MIN()`, `MAX()` — работают как обычные агрегаты, но в пределах окна

- **Функции смещения (Offset functions):**  
  `LAG()` — значение из предыдущей строки  
  `LEAD()` — значение из следующей строки  
  `FIRST_VALUE()` — первое значение в окне  
  `LAST_VALUE()` — последнее значение в окне

**Пример использования разных групп:**

    SELECT 
        employee_id,
        department_id,
        salary,
        ROW_NUMBER() OVER w AS row_num,
        SUM(salary) OVER w AS sum_sal,
        LAG(salary) OVER w AS prev_salary
    FROM employees
    WINDOW w AS (PARTITION BY department_id ORDER BY salary DESC)
    ORDER BY department_id, salary DESC;

**Результат:**

| employee_id | department_id | salary | row_num | sum_sal | prev_salary |
|------------|---------------|--------|---------|---------|-------------|
| 101        | 1             | 100000 | 1       | 250000  | NULL        |
| 102        | 1             | 80000  | 2       | 250000  | 100000      |
| 103        | 1             | 70000  | 3       | 250000  | 80000       |
| 201        | 2             | 120000 | 1       | 210000  | NULL        |
| 202        | 2             | 90000  | 2       | 210000  | 120000      |

Запрос нумерует сотрудников внутри каждого департамента по убыванию зарплаты, показывает общую сумму зарплат по департаменту и зарплату предыдущего сотрудника. Ключевое слово `WINDOW` позволяет задать окно один раз и использовать его в нескольких функциях.

### Хранимые процедуры, функции, триггеры

- **Хранимая процедура** — набор SQL-команд, хранящийся на сервере. Может выполнять несколько операций, менять данные, но не обязана возвращать значение (в отличие от функции).  
  *Пример:* `CALL update_prices(10)`

- **Функция** — возвращает значение, может использоваться в запросах (например, в `SELECT`).  
  *Пример:* `SELECT calculate_tax(price) FROM products`

- **Триггер** — автоматически выполняет код при событии (`INSERT`, `UPDATE`, `DELETE`). Используется для валидации, аудита, поддержки целостности данных.  
  *Пример:* при добавлении заказа автоматически обновить остаток товара

**В Python** (SQLAlchemy) триггеры создаются через DDL или событийную систему `event.listen`.

#### Инициализация

    engine = create_async_engine('postgresql+asyncpg://user:pass@localhost/db')
    AsyncSessionLocal = async_sessionmaker(engine, expire_on_commit=False)

    # в запросе
    async def get_db():
        async with AsyncSessionLocal() as session:

[Dev.to](https://dev.to/akarshan/asynchronous-database-sessions-in-fastapi-with-sqlalchemy-1o7e)

**Зачем движок, сессия, коннекшн?**  
- **Engine (движок):** создаётся один на всё приложение. Знает, как подключиться к БД (URL, пул соединений, диалект SQL).
- **Connection (соединение):** непосредственно канал связи с БД. Берется из пула, который поддерживает Engine.
- **Session (сессия):** работает поверх connection. Отслеживает изменения объектов, управляет транзакциями, сохраняет объекты в памяти. Это то, с чем вы работаете в коде.
- **Фабрика сессий:** нужна, чтобы в каждом запросе создавать новую сессию — с пустым кэшем, без "прилипших" старых объектов. Если использовать одну сессию на всё приложение, объекты будут накапливаться в памяти и могут устаревать.

#### Типы результатов запросов

**`session.execute()` возвращает объект `Result`.** Он содержит строки, но напрямую не итерируется. Чтобы получить данные, используем методы `Result`.

**Основные методы `Result`:**

- **`scalar()`** — возвращает первый элемент первой строки. Если запрос возвращает несколько колонок, берётся только первая.
- **`scalars()`** — возвращает итератор, извлекая первый элемент из каждой строки. `scalars().all()` даёт список первых колонок всех строк.
- **`all()`** — возвращает список всех строк целиком (каждая строка — кортеж `Row` со всеми колонками). Доступ к колонкам — по индексу: `row[0]`, `row[1]`.
- **`mappings().all()`** — возвращает список строк в виде словарей. Доступ к колонкам — по имени: `row["id"]`, `row["name"]`. Удобно, когда много колонок или одинаковые имена.
  
      stmt = select(Rooms.id, Rooms.name)
      result = await session.execute(stmt)
      rows = result.mappings().all()
      print(rows[0]["name"])  # "Комната 101"

  Другие методы `mappings()`: `first()`, `one()`, `scalar()` — работают аналогично, но возвращают словари.

**Для ленивой загрузки (не грузить всё сразу в память):**

- **`stream_scalars()`** — асинхронный итератор для случаев, когда нужна только первая колонка. Каждая итерация асинхронно загружает следующую строку, не блокируя событийный цикл.
- **`execute() + yield_per`** — универсальный способ для любых запросов с контролем размера чанка. Первый чанк загружается асинхронно, последующие — синхронно внутри цикла.

**Примеры:**

**1. `select(Rooms)`**

    # Каждая строка содержит объект Rooms
    r1 = scalar()           # Rooms (первая строка, первая колонка)
    r2 = scalars().all()    # List[Rooms] (из каждой строки первая колонка)
    r3 = all()              # List[Row], где каждая строка содержит (Rooms,)
    r3[0].Rooms             # доступ к объекту Rooms из первой строки (r3)

**2. `select(Rooms.id)`**

    # Каждая строка содержит одно число (id)
    r1 = scalar()           # int (первый id)
    r2 = scalars().all()    # List[int] (все id)
    r3 = all()              # List[Row] с одной колонкой id
    r3[0].id                # доступ к id из первой строки (r3)

**3. `select(Rooms.id, Rooms.name, Hotels.name).join(Hotels)`**

    # Каждая строка содержит три колонки: id комнаты, название комнаты, название отеля
    r = all()               # List[Row]
    r[0].id                 # id комнаты
    r[0].name               # название комнаты
    r[0].name_1             # название отеля (второе name переименовано в name_1)

**4. `select(Rooms, Hotels).join(Hotels)`**

    # Каждая строка содержит два объекта: Rooms и Hotels
    r = all()               # List[Row]
    r[0].Rooms              # объект Rooms из первой строки
    r[0].Hotels             # объект Hotels из первой строки
    r[0].Rooms.name         # название комнаты
    r[0].Hotels.name        # название отеля

**5. `select(Hotels).options(selectinload(Hotels.rooms))`**

    # Загружаем отели с комнатами (дополнительный запрос)
    r = scalars().all()     # List[Hotels]
    r[0].rooms              # список комнат первого отеля (уже загружен)
    r[0].rooms[0].name      # название первой комнаты

**6. Ленивая загрузка с контролем памяти (execute + yield_per)**

    from sqlalchemy.ext.asyncio import AsyncSession

    async def fetch_rows(session: AsyncSession, stmt, chunk_size=100):
        """Универсальный генератор с контролем размера чанка.
        Первый чанк загружается асинхронно, последующие — синхронно."""
        result = await session.execute(stmt.execution_options(yield_per=chunk_size))
        for row in result:  # начиная со второго чанка — синхронно
            yield row

    # Использование:
    stmt = select(Rooms, Hotels).join(Hotels)
    async for row in fetch_rows(session, stmt, 50):
        print(row.Rooms.name, row.Hotels.name)

**7. stream_scalars() — полностью асинхронная итерация**

    stmt = select(Rooms)
    result = await session.stream_scalars(stmt)  # асинхронный итератор
    async for room in result:  # каждая итерация асинхронно ждёт следующую строку
        print(room.name)       # не блокирует событийный цикл

### Транзакции и изоляция

**Транзакция** — группа операций, которая выполняется как единое целое: либо все операции применяются (COMMIT), либо ни одной (ROLLBACK). Обеспечивает свойства ACID:

- **A**tomicity (атомарность) — транзакция выполняется целиком или не выполняется вообще
- **C**onsistency (согласованность) — транзакция переводит данные из одного согласованного состояния в другое
- **I**solation (изоляция) — параллельные транзакции не влияют друг на друга
- **D**urability (долговечность) — после COMMIT изменения сохраняются даже при сбое

**Проблемы параллельного доступа:**

- **«Грязное» чтение (Dirty Read)** — чтение данных, изменённых незавершённой транзакцией (которая потом может откатиться)
- **Неповторяющееся чтение (Non-repeatable Read)** — в рамках одной транзакции повторное чтение той же строки даёт разные значения, потому что другая транзакция изменила и закоммитила эту строку
- **Фантомное чтение (Phantom Read)** — в рамках одной транзакции повторный запрос возвращает разное количество строк, потому что другая транзакция добавила или удалила строки, удовлетворяющие условию

**Уровни изоляции транзакций** («+» — предотвращает, «-» — не предотвращает)

| Уровень изоляции | Грязное чтение | Неповторяющееся чтение | Фантомное чтение |
|------------------|:--------------:|:----------------------:|:-----------------:|
| **Read Uncommitted** | - | - | - |
| **Read Committed**   | + | - | - |
| **Repeatable Read**  | + | + | - |
| **Serializable**     | + | + | + |

*Примечание: потерянные обновления стандарт SQL не регламентирует — в разных СУБД могут защищать от них и на более низких уровнях.*

[Wikipedia](https://ru.wikipedia.org/wiki/Уровень_изолированности_транзакций)

**Реализация в СУБД**

**PostgreSQL:**
- **Read Uncommitted** — не поддерживается, работает как Read Committed
- **Read Committed** (по умолчанию) — каждый запрос видит только закоммиченные данные на момент своего начала
- **Repeatable Read** — используется snapshot isolation (снимок данных на момент первого запроса). Предотвращает грязное и неповторяющееся чтение, но фантомы возможны
- **Serializable** — через SSI (Serializable Snapshot Isolation) отслеживает конфликты и прерывает транзакции, которые могли бы нарушить сериализуемость. Транзакции выполняются так, как будто идут одна за другой

**MySQL (InnoDB):**
- **Read Uncommitted** — можно читать незакоммиченные данные
- **Read Committed** — каждый запрос видит только закоммиченные данные на момент своего начала
- **Repeatable Read** (по умолчанию) — используется snapshot isolation (снимок данных на момент первого чтения). Предотвращает грязное и неповторяющееся чтение, а также фантомы — за счёт gap locks (блокировка не только строк, но и промежутков между ними)
- **Serializable** — транзакции выполняются так, как будто идут одна за другой (через блокировки таблиц или более строгие блокировки диапазонов)

**Важно:** в MySQL на уровне Repeatable Read фантомы предотвращаются, хотя стандарт SQL этого не требует. В PostgreSQL на этом уровне фантомы возможны. Таблицу выше нужно читать как описание стандарта, а не конкретных СУБД.

**Транзакции в SQLAlchemy (пример с уровнем изоляции)**  

    from sqlalchemy import create_engine
    from sqlalchemy.orm import sessionmaker

    engine = create_engine('postgresql://...', isolation_level='REPEATABLE READ')
    Session = sessionmaker(bind=engine)

    with Session.begin() as session:
        user = session.query(User).filter_by(id=1).with_for_update().first()
        user.balance += 100
        # при выходе из with — commit, при ошибке — rollback

### ltree (древовидные структуры в PostgreSQL)

**Что такое:** расширение PostgreSQL для хранения и поиска путей в дереве. Путь задаётся строкой с точками, например: `"electronics.phones.apple"`.

**Когда использовать:** иерархии категорий товаров, организационная структура, древо комментариев, географическая иерархия.

**Сравнение с альтернативами:**

| Подход | Хранение | Поиск поддерева | Простота |
|---|---|---|---|
| ltree | `"electronics.phones.apple"` | Быстрый (индекс) | Высокая |
| Parent ID | `parent_id` ссылается на ту же таблицу (как id → parent_id) | Медленный (рекурсивный CTE: `WITH RECURSIVE`) | Средняя |
| Closure table | Отдельная таблица связей (все пути между узлами) | Быстрый | Низкая (нужно поддерживать связи) |

**Включение и создание таблицы:**

    CREATE EXTENSION IF NOT EXISTS ltree;

    CREATE TABLE categories (
        id SERIAL PRIMARY KEY,
        name TEXT,
        path LTREE
    );

**Индексы:**

    CREATE INDEX idx_categories_gin ON categories USING GIN (path);   -- быстрее для поиска поддеревьев (<@, @>)
    CREATE INDEX idx_categories_gist ON categories USING GIST (path); -- поддерживает поиск по шаблону (~)

**Операторы:**

| Оператор | Описание | Пример | Результат |
|---|---|---|---|
| `<@` | Левое — предок правого? | `'electronics' <@ 'electronics.phones'` | true |
| `@>` | Левое — потомок правого? | `'electronics.phones' @> 'electronics'` | true |
| `~` | Соответствует ли пути шаблону? | `'electronics.phones' ~ '*.phones'` | true |

**Типовые запросы:**

    -- Все подкатегории (включая текущую)
    -- Результат: electronics.phones, electronics.phones.apple, electronics.phones.samsung
    SELECT * FROM categories WHERE path <@ 'electronics.phones';

    -- Предки (хлебные крошки)
    -- Результат: electronics, electronics.phones, electronics.phones.apple
    SELECT * FROM categories WHERE path @> 'electronics.phones.apple';

    -- Категории на глубине 2 от корня
    -- Результат: electronics.phones, electronics.laptops, electronics.gpus
    SELECT * FROM categories WHERE path ~ '*.*';

**В SQLAlchemy:**

    from sqlalchemy.dialects.postgresql import LTREE

    class Category(Base):
        __tablename__ = 'categories'
        id = Column(Integer, primary_key=True)
        name = Column(String)
        path = Column(LTREE)

    # Поиск подкатегорий
    subcategories = session.query(Category).filter(
        Category.path.op('<@')('electronics.phones')
    ).all()

## 6. Базы данных (NoSQL)

### Классификация и принципы работы

**Ключ-значение (Key-Value)** — Redis, Memcached  
Хранят данные в оперативной памяти как огромный словарь. Доступ к значению по ключу за микросекунды — не нужно ходить на диск, парсить SQL, строить планы запросов. Идеально для кэша, сессий, счётчиков.

**Документные (Document)** — MongoDB, CouchDB  
Хранят JSON-подобные документы. Вся связанная информация (например, заказ с товарами) лежит в одном документе, а не размазана по таблицам. Не нужно делать JOIN — прочитал документ и получил всё сразу.

**Колоночные (Wide-Column)** — Cassandra, HBase, ClickHouse  
Данные хранятся не строками, а колонками: все значения одной колонки лежат подряд непрерывным блоком. Для запроса `AVG(salary)` читается только блок с зарплатами, остальные колонки не трогаются. Данные в колонке однотипные — отлично сжимаются. Можно читать колонку сразу большим куском, экономя время на ввод-вывод.

**Графовые (Graph)** — Neo4j, ArangoDB  
Хранят узлы и связи между ними как отдельные сущности с указателями. Запросы "друзья друзей" или "кратчайший путь" выполняются за миллисекунды, проходя по связям. В SQL пришлось бы делать десятки JOIN.

### Конкретные БД и библиотеки для Python

- **Redis** — in-memory хранилище структур данных (строки, списки, хэши, множества). Используется для кэша, сессий, pub/sub.  
  *Библиотеки:* `redis-py` (синхронный и асинхронный клиент)

- **Memcached** — простое in-memory хранилище ключ-значение. Функционально беднее Redis (только строки), но часто быстрее для простого кэширования.  
  *Библиотеки:* `pymemcache`, `python-memcached`

- **MongoDB** — документная БД (BSON). Гибкая схема, горизонтальное масштабирование через шардинг.  
  *Библиотеки:* `beanie` (ODM с asyncio), `motor` (асинхронный драйвер), `pymongo` (синхронный драйвер)

- **Elasticsearch** — поисковый движок на базе Lucene. Хранит данные в инвертированных индексах. Используется для полнотекстового поиска и аналитики по логам (ELK).  
  *Библиотеки:* `elasticsearch-py` (официальный клиент)

- **Tarantool** — in-memory БД и сервер приложений. Поддерживает вторичные индексы, хранимые процедуры на Lua.  
  *Библиотеки:* `tarantool-connector`

- **ClickHouse** — колоночная аналитическая БД от Яндекс. Оптимизирована для OLAP-запросов (агрегации по большим объёмам). Оконные функции поддерживаются с версии 21.4 (с ограничениями).  
  *Библиотеки:* `clickhouse-driver`, `clickhouse-connect`

## 7. Брокеры сообщений и стриминг

**Брокеры сообщений (Message Brokers)** — обеспечивают асинхронное взаимодействие сервисов через очереди.

- **RabbitMQ:** поддерживает AMQP. Работает по модели "обменник (exchange) → очередь".  
  - **Exchange:** получает сообщение от продюсера и маршрутизирует в очереди по правилам (binding) и типу (direct, topic, fanout, headers).  
  - RabbitMQ гарантирует доставку, подтверждения, dead letter exchanges.  
  - В Docker достаточно одного контейнера, есть веб-интерфейс.

- **Apache Kafka:** платформа для стриминга событий (event streaming). В отличие от RabbitMQ, хранит поток событий в **логе** и позволяет перечитывать его.  
  - Высокая пропускная способность, отказоустойчивость, репликация.  
  - Используется для сбора логов, трекинга активности, CDC (Change Data Capture).  
  - Настройка сложнее: нужны Kafka (брокеры), Zookeeper (или KRaft), иногда Kafka Connect. Обычно 3+ контейнера.

### Потоковая передача данных (Streaming) — разбор "каши"

| Технология | Тип | Направление | Пример использования |
| :--- | :--- | :--- | :--- |
| **SSE (Server-Sent Events)** | HTTP-стриминг | Сервер → Клиент | Уведомления, лента новостей (односторонние) |
| **WebSockets** | Дуплексный протокол | Двустороннее | Чаты, онлайн-игры, биржевые котировки |
| **Centrifugo** | Готовый сервер | Двустороннее | Упрощает использование WebSockets/SSE, интеграция с бэкендом |
| **RabbitMQ** | Брокер очередей | Между сервисами | Асинхронные задачи (Celery), гарантированная доставка |
| **Kafka** | Платформа стриминга | Между сервисами | Событийная шина, обработка потоков данных, переигрывание событий |
| **Потоковое видео** | RTMP, HLS, WebRTC | Двустороннее/одностор. | Прямые эфиры, видеозвонки (это отдельный стек, не путать с сообщениями) |
| **I/O Streams** (файлы, память) | Потоки ОС | Внутри процесса | Чтение/запись файлов, pipe-ы между процессами |

## 8. Big Data и ETL

**ETL (Extract, Transform, Load)** — процесс извлечения данных из источников, преобразования и загрузки в хранилище (Data Warehouse).

**Hadoop** — экосистема для распределенного хранения и обработки больших данных:  
- **HDFS (Hadoop Distributed File System)** — распределенная файловая система.  
- **YARN** — менеджер ресурсов.  
- **MapReduce** — модель обработки (устаревает).

**Apache Hive** — хранилище данных поверх Hadoop. Запросы на SQL-подобном языке HiveQL.

**Apache Spark** — фреймворк для обработки больших данных в памяти (быстрее MapReduce). Поддерживает пакетную обработку, стриминг, машинное обучение (MLlib), графы (GraphX). Может работать поверх Hadoop или отдельно.

**Хранилища данных (Data Warehouse)**  
- **Схема "Звезда" (Star schema):** одна таблица фактов (facts) в центре, окруженная таблицами измерений (dimensions). Денормализована для быстрых агрегатов.  
- **Схема "Снежинка" (Snowflake schema):** звезда, где измерения дополнительно нормализованы.

## 9. Веб-фреймворки и API

### Стили API

- **REST API:** архитектурный стиль на базе HTTP. Ресурсы, методы (GET, POST, PUT, PATCH, DELETE), stateless. [Skillbox](https://skillbox.ru/media/code/rest-api-chto-eto-takoe-i-kak-rabotaet/)  
- **SOAP:** протокол на XML. Тяжеловесный, строгая спецификация.
- **GraphQL:** язык запросов, позволяющий клиенту запрашивать только нужные данные. Решает проблемы:
  - over-fetching — получение лишних данных, которые не нужны клиенту (REST часто отдаёт весь объект целиком)
  - under-fetching — нехватка данных в одном запросе, приходится делать несколько (REST может требовать цепочку запросов)

### HTTP Методы

- **GET** — получить ресурс.  
- **POST** — создать новый.  
- **PUT** — полностью заменить.  
- **PATCH** — частично изменить.  
- **DELETE** — удалить.  
- **HEAD** — как GET, но без тела ответа (только заголовки).  
- **OPTIONS** — узнать доступные методы для ресурса. [StackOverflow](https://stackoverflow.com/questions/27030649/explain-and-example-about-get-delete-post-put-options-patch)

**Заголовки HTTP** — метаданные запроса/ответа (Content-Type, Authorization, Cache-Control).

### ASGI, WSGI и серверы

- **WSGI (Web Server Gateway Interface)** — синхронный стандарт для Python-приложений (Django, Flask). Серверы: Gunicorn, uWSGI.  
- **ASGI (Asynchronous Server Gateway Interface)** — асинхронный стандарт (FastAPI, Django с ASGI). Серверы: Uvicorn, Daphne.  
- **Nginx** — не сервер приложений, а **обратный прокси**. Стоит перед Gunicorn/Uvicorn: принимает запросы из интернета, отдает статику, балансирует, терминирует SSL, а динамику проксирует на бэкенд.

### Веб-фреймворки (сравнение)

- **Django:** монолит, батарейки включены (ORM, админка, аутентификация). Большое сообщество, много готовых решений. Тяжелый, но быстрый в разработке. Раньше только WSGI, с 3.x — ASGI (асинхронные представления, но ORM пока синхронная).  
- **Flask:** микрофреймворк, синхронный, WSGI. Гибкий, минималистичный, многое подключается расширениями. Сообщество большое, производительность выше Django за счет легкости.  
- **FastAPI:** современный асинхронный фреймворк (ASGI), автоматическая документация (OpenAPI), валидация через Pydantic. Нативная поддержка `async/await`. Очень производительный (на уровне Node.js). Сообщество растет, но меньше, чем у Django/Flask.

### Real-time коммуникация

- **WebSockets:** полнодуплексная связь поверх TCP. В FastAPI поддержка из коробки (быстрее, чем в Django/Flask).  
- **SSE (Server-Sent Events):** односторонняя связь сервер→клиент по HTTP. Проще WebSockets. [Tproger](https://tproger.ru/articles/sse-vs-websockets--v-chem-raznica-)  
- **Centrifugo:** сервер для real-time сообщений. Бэкенд публикует события, клиенты получают через WebSockets/SockJS/SSE. [Centrifugo docs](https://centrifugal.dev/)

### Тестирование FastAPI с БД

[StackOverflow](https://stackoverflow.com/questions/67255653/how-to-set-up-and-tear-down-a-database-between-tests-in-fastapi) — создание тестовой БД, миграции, фикстуры.

## 10. Архитектура и паттерны

### Группа А: Как писать код (уровень реализации)

#### Базовые принципы

- **DRY (Don't Repeat Yourself)** — избегай дублирования кода, выноси общее в функции/классы.

- **KISS (Keep It Simple, Stupid)** — чем проще решение, тем лучше, не усложняй без необходимости.

- **YAGNI (You Ain't Gonna Need It)** — не пиши код "на всякий случай", добавляй функциональность только когда она реально понадобилась.

- **Композиция vs Наследование** — предпочитай композицию наследованию. Наследование создаёт жёсткую связь: изменение родителя может сломать потомков. Композиция гибче: объекты можно заменять, переиспользовать и тестировать изолированно.

  *Наследование "is-a":* машина *является* двигателем? Нелогично. К тому же нельзя заменить двигатель без изменения класса: `class Car(Engine):`

  *Композиция "has-a":* машина *имеет* двигатель. Двигатель можно передать извне, подменить, замокать в тестах: `class Car: def __init__(self, engine): self.engine = engine`

#### SOLID

Пять принципов, которые помогают делать классы гибкими и удобными для поддержки. SOLID **используют** на этапе детальной реализации: как организовать класс, чтобы при добавлении новой функциональности не пришлось переписывать существующий код.

- **Single Responsibility** — у класса должна быть одна причина для изменения.  
  *Пример:* если класс `User` хранит данные, сохраняет их в БД и отправляет письма — лучше разделить на `User`, `UserRepository` и `UserNotifier`.

- **Open/Closed** — классы открыты для расширения, но закрыты для изменения.  
  *Пример:* вместо изменения класса добавить наследника или стратегию.

- **Liskov Substitution** — наследники должны дополнять, а не ломать поведение предков.  
  *Пример:* если метод родителя бросает `ValueError`, наследник не должен бросать `Exception`.

- **Interface Segregation** — много специализированных интерфейсов лучше одного общего.  
  *Пример:* вместо `Worker` с методами `work()` и `eat()` разделить на `Workable` и `Eatable`.

- **Dependency Inversion** — модули верхних уровней не зависят от нижних; оба зависят от абстракций.  
  *Пример:* сервис зависит от интерфейса репозитория, а не от конкретной БД.

#### GRASP

Принципы назначения ответственности классам. GRASP **используют** на этапе распределения обязанностей: какой класс за что отвечает. Например, должен ли заказ сам считать свою сумму, или для этого нужен отдельный класс?

- **Информационный эксперт (Information Expert)** — назначать ответственность классу, который обладает необходимой информацией.  
  *Пример:* `calculate_total()` в `Order`, а не в отдельном калькуляторе.

- **Создатель (Creator)** — класс B должен создавать экземпляры класса A, если B содержит A, агрегирует A, использует A или знает его начальные данные.  
  *Пример:* `Order` создаёт `OrderItem`.

- **Низкая связанность (Low Coupling)** — классы должны зависеть друг от друга как можно меньше.  
  *Пример:* класс `ReportGenerator` принимает интерфейс `DataSource`, а не конкретную БД. Это позволяет подменить источник данных (файл, API) без изменения генератора отчётов.

- **Высокое зацепление (High Cohesion)** — класс отвечает за одну чёткую задачу: все его методы направлены на решение этой задачи.  
  *Пример:* `UserRepository` — только БД, `UserNotifier` — только уведомления.

- **Полиморфизм (Polymorphism)** — использовать полиморфизм вместо `if/else`.  
  *Пример:* `animal.sound()` вместо проверки типа.

- **Чистое изготовление (Pure Fabrication)** — класс, которого нет в предметной области, но который вводится для снижения связанности или повышения зацепления.  
  *Пример:* `UserService` — не существует в реальном мире как объект, но нужен для организации бизнес-логики.

- **Посредник (Indirection)** — вводить промежуточный объект между компонентами.  
  *Пример:* слой репозитория между сервисом и БД.

- **Устойчивость к изменениям (Protected Variations)** — защищать компоненты через интерфейсы.  
  *Пример:* фасад `EmailSender` скрывает детали работы с почтовым сервером. При замене почтового сервера меняется только фасад.

- **Переадресация (Controller)** — обработка системных событий выносится в отдельный объект, который не содержит логики, а только вызывает нужные сервисы и формирует ответ.  
  *Пример:* в FastAPI контроллер — это функция-обработчик роута: получает запрос, вызывает сервис, возвращает JSON.

#### Техники программирования
Конкретные приёмы реализации, которые помогают применять принципы на практике.

- **Guard Clause (охранное выражение)** — проверяй условия в начале функции и сразу выходи, чтобы избежать глубокой вложенности if.


    def process(data):
        if not data:
            return None  # guard clause
        # основная логика
        return result

- **Dependency Injection** — передавай зависимости извне, а не создавай внутри. Вместо `self.db = Database()` внутри класса получаем `db` через параметры конструктора. Упрощает тестирование и делает код гибче.

- **Law of Demeter (Принцип наименьшего знания)** — объект должен знать только о своих непосредственных соседях. Нельзя вызывать методы у объектов, полученных через цепочку вызовов.

  *Непосредственные соседи:* свои поля, аргументы методов, объекты, которые создал сам.

  Плохо (цепочка):
  
      user.getOrder().getItems().getTotal()
  
  Хорошо (прямой запрос):
  
      user.getOrderTotal()
  
  В хорошем варианте User знает только про Order, а Order сам разбирается со своими Items. Если Items изменится, User не придётся править.

- **Репозиторий (Repository)** — прослойка между бизнес-логикой и хранилищем данных, создаёт иллюзию работы с коллекцией объектов в памяти. Позволяет заменить БД, не меняя код выше.

- **Unit of Work (UoW)** — отслеживает изменения объектов и управляет транзакциями, чтобы сохранить всё согласованно за один раз (аналог Session в SQLAlchemy).

  *Суть:* UoW накапливает изменения в объектах, а при вызове `commit()` сам генерирует нужные SQL-запросы и отправляет их все одной пачкой в рамках одной транзакции.

  Плохо (ручное управление транзакцией с отдельными запросами):

      db.begin()
      try:
          db.execute("INSERT INTO users ...")   # запрос 1
          db.execute("INSERT INTO orders ...")  # запрос 2  
          db.execute("INSERT INTO items ...")   # запрос 3
          db.commit()
      except:
          db.rollback()

  *Что здесь происходит:* три отдельных запроса летят в БД по очереди. Каждый запрос — отдельное обращение к БД.

  Хорошо (UoW с группировкой):

      with uow:
          uow.register_new(user)    # объект в памяти
          uow.register_new(order)   # объект в памяти
          uow.register_new(items)   # объект в памяти
          # при выходе из with UoW отправляет в БД один пакет:
          # INSERT INTO users ...; INSERT INTO orders ...; INSERT INTO items ...;

  *Разница:* при ручном управлении БД получает команды по одной (3 обращения). При UoW — все три команды одним пакетом (1 обращение). В обоих случаях транзакция одна.

  **Дополнительная возможность UoW:** оптимизация запросов. Если один и тот же объект изменили несколько раз до commit(), UoW отправит только последнее состояние, а не все промежуточные.

#### Паттерны проектирования (GoF — Gang of Four)

Типовые решения для часто встречающихся задач в объектно-ориентированном программировании.

**Почему Gang of Four:** так называют четверых авторов книги "Design Patterns: Elements of Reusable Object-Oriented Software" (1994) — Эриха Гамму, Ричарда Хелма, Ральфа Джонсона и Джона Влиссидеса. Книга систематизировала 23 паттерна.

- **Порождающие:**
  
  `Singleton` — гарантирует один экземпляр класса. *Где применяется:* логгер, пул соединений, кэш.
  
  `Factory Method` — метод, создающий объекты (вместо прямого вызова конструктора). *Где применяется:* создание разных типов документов в зависимости от входных данных.
  
  `Abstract Factory` — фабрика для создания **семейств связанных объектов** (например, все элементы интерфейса для Windows: кнопка + чекбокс + скролл — в одном стиле).
  
  `Builder` — пошаговое создание сложного объекта. *Где применяется:* построитель SQL-запроса.

- **Структурные:**
  
  `Adapter` — позволяет объектам с несовместимыми интерфейсами работать вместе. *Где применяется:* обёртка для сторонней библиотеки с неудобным интерфейсом.
  
  `Proxy` — объект, контролирующий доступ к другому объекту. Прокси в кэшировании — объект, который стоит перед реальной БД (или любым тяжёлым ресурсом) и перехватывает повторные вызовы, возвращая сохранённый результат.
  
  `Decorator` — динамически добавляет объекту новую функциональность. *Где применяется:* добавление сжатия или шифрования к потоку данных.
  
  `Facade` — предоставляет простой способ работы со сложной системой, скрывая детали взаимодействия её компонентов.
  
        class CPU:
            def freeze(self): print("CPU: freeze")
  
        class Memory:
            def load(self, data): print(f"Memory: load {data}")
  
        class HardDrive:
            def read(self): return "data"
  
        class Computer:  # Фасад
            def __init__(self):
                self.cpu = CPU()
                self.memory = Memory()
                self.hard_drive = HardDrive()
            
            def start(self):
                self.cpu.freeze()
                self.memory.load(self.hard_drive.read())
  
        Computer().start()  # один метод вместо трёх

- **Поведенческие:**
  
  `Iterator` — описывает идею обхода коллекции без доступа к её внутренностям. *Где применяется:* встроенные итераторы Python.
  
  `Observer` — механизм подписки: объект уведомляет подписчиков об изменениях.
  
        class NewsChannel:
            def __init__(self):
                self.subscribers = []
            
            def subscribe(self, subscriber):
                self.subscribers.append(subscriber)
            
            def notify(self, news):
                for sub in self.subscribers:
                    sub.update(news)
  
        class Subscriber:
            def update(self, news):
                print(f"Получено: {news}")
  
        channel = NewsChannel()
        sub = Subscriber()
        channel.subscribe(sub)
        channel.notify("Срочные новости!")  # подписчик получит уведомление
  
  `Strategy` — взаимозаменяемые алгоритмы.
  
        class QuickSort:
            def sort(self, data): return sorted(data)
  
        class MergeSort:
            def sort(self, data): return sorted(data)
  
        class DataProcessor:
            def __init__(self, strategy):
                self._strategy = strategy
            
            def set_strategy(self, strategy):
                self._strategy = strategy
            
            def process(self, data):
                return self._strategy.sort(data)
  
        processor = DataProcessor(QuickSort())
        print(processor.process([3, 1, 4]))  # быстрая сортировка
        processor.set_strategy(MergeSort())
        print(processor.process([3, 1, 4]))  # сортировка слиянием
  
  `Command` — инкапсулирует запрос как объект, позволяя сохранять его и выполнять позже.
  
        class Command:
            def __init__(self, action):
                self.action = action
            def execute(self):
                self.action()
  
        def turn_on(): print("Свет включён")
        def turn_off(): print("Свет выключен")
  
        # Команды можно сохранять в истории и выполнять позже
        history = [Command(turn_on), Command(turn_off)]
        for cmd in history:
            cmd.execute()

### Группа Б: Как организовать систему (архитектура)

#### Архитектурные стили

- **MVC (Model-View-Controller)** — разделение данных (Model), отображения (View) и логики управления (Controller).

- **MTV (Model-Template-View)** — вариант MVC в Django. Путаница в терминах:

  | Роль | В MVC | В Django (MTV) |
      |-------|--------|------------------|
      | Данные | Model | Model |
      | Отображение (UI) | View | Template |
      | Управление | Controller | View |

- **Событийно-ориентированная архитектура (Event-Driven)** — система строится вокруг событий, сервисы общаются через события, а не прямые вызовы.

- **Монолит vs Микросервисы** — монолит проще в разработке и отладке, микросервисы масштабируются независимо и позволяют использовать разные технологии.

- **Паттерн Душитель (Strangler Pattern)** — постепенная замена частей работающей системы, пока старая система продолжает функционировать.

- **Микросервисные паттерны:**
  
  - **Сага (Saga)** — управление распределёнными транзакциями через последовательность локальных транзакций. Если одна транзакция падает — запускаются компенсирующие действия для предыдущих.
  
    *Хореография (Choreography):* каждый сервис публикует события, другие сервисы на них подписываются и выполняют свои действия. Нет центрального координатора.
    
        Сервис Заказов: создал заказ → событие OrderCreated
        Сервис Платежей: получил событие → списал деньги → событие PaymentProcessed
        Сервис Доставки: получил событие → оформил доставку
    
    *Оркестрация (Orchestration):* центральный координатор (оркестратор) говорит сервисам, что делать.
    
        Оркестратор: "Заказ, создайся"
        Оркестратор: "Платеж, спиши деньги"  
        Оркестратор: "Доставка, оформи"

  - **Circuit Breaker** — предотвращает повторные вызовы к падающему сервису; при превышении порога ошибок цепь "размыкается" и вызовы сразу возвращают ошибку или fallback.

  - **Bulkhead** — изоляция ресурсов (пулы соединений, потоки) по принципу отсеков на корабле, чтобы сбой в одном компоненте не убил всю систему.

#### Domain-Driven Design (DDD)
Подход к проектированию сложных предметных областей, где код отражает реальные бизнес-процессы.

- **Агрегаты и границы согласованности** — группа объектов, с которыми работают как с единым целым; изменения всегда применяются ко всему агрегату в одной транзакции.  
  *Пример:* заказ с позициями — нельзя изменить позицию, не затрагивая итоговую сумму заказа. Всё это один агрегат.

- **Сервисный слой (Application Service)** — координирует выполнение Use Case'ов, оркестрирует доменные объекты, не содержит бизнес-логики.  
  *Пример:* сервис оформления заказа вызывает методы репозитория, проверяет права, но не решает, какой товар можно заказать.

- **Доменные события (Domain Events)** — оповещают другие части системы о том, что произошло что-то важное. Шина сообщений доставляет события подписчикам.  
  *Пример:* после оформления заказа публикуется событие `OrderPlaced`, на которое подписан сервис уведомлений.

#### CQRS и Event Sourcing
Специализированные подходы для систем с высокой нагрузкой или требованием к аудиту.

- **CQRS (Command Query Responsibility Segregation)** — разделение операций чтения (Queries) и записи (Commands) на разные модели, иногда даже на разные базы.  
  *Пример:* заказы пишутся в основную БД, а для отчётов используется отдельная read-база, оптимизированная под выборки.

- **Event Sourcing** — хранение состояния как последовательности событий, а не как текущего среза. Текущее состояние восстанавливается воспроизведением событий.  
  *Пример:* вместо хранения текущего баланса счёта храним все операции (пополнения, списания). Баланс считается как сумма операций.

### Группа В: Как организовать процесс (методологии)

- **TDD (Test-Driven Development)** — разработка через тестирование: сначала пишется тест, потом код, который сделает тест зелёным, потом рефакторинг. Гарантирует, что код покрыт тестами и делает архитектуру более гибкой.

## 11. Инструменты и инфраструктура

### Poetry

Управление зависимостями и сборка пакетов. Альтернатива pip + venv + setup.py.

**Основные команды:**
- `poetry new my-lib` — создать проект
- `poetry add requests` — добавить зависимость
- `poetry add --dev pytest` — dev-зависимость
- `poetry shell` — активировать виртуальное окружение
- `poetry build` — собрать пакет
- `poetry publish` — опубликовать в PyPI

**Где создаётся виртуальное окружение:**
- По умолчанию: в отдельном каталоге вне проекта  
  (`~/Library/Caches/pypoetry` на macOS,  
  `~/.cache/pypoetry` на Linux,  
  `%APPDATA%\pypoetry\Cache` на Windows)
- `poetry config virtualenvs.in-project true` — создавать `.venv` внутри папки проекта (как `python -m venv`)

**Отличие от pip:** Poetry автоматически создаёт виртуальное окружение, управляет версиями (lock-файл), разделяет зависимости на основные и dev.

### Линтеры, форматтеры и статические анализаторы

- **black** — форматтер. Приводит код к единому виду по правилам PEP 8 (отступы, пробелы, длина строк). Сообщает и исправляет.
- **isort** — форматтер. Сортирует импорты (стандартные, сторонние, свои). Сообщает и исправляет.
- **flake8** — линтер. Проверяет стиль (PEP 8), ищет ошибки, проверяет докстринги (PEP 257 — оформление документации). Сообщает, но не исправляет.
- **pylint** — линтер. Проверяет стиль, ошибки, "запахи кода" (code smells — потенциальные проблемы в дизайне) и докстринги (PEP 257). Сообщает, но не исправляет.
- **mypy** — статический анализатор типов. Проверяет соответствие кода аннотациям (PEP 484). Сообщает, но не исправляет.

### Git / CI/CD

**Основные команды (база):**
- `git init` — создать репозиторий
- `git clone <url>` — скопировать удалённый репозиторий
- `git add <file>` — добавить файл в индекс (подготовить к коммиту)
- `git commit -m "message"` — закоммитить изменения из индекса
- `git push` — отправить коммиты на удалённый репозиторий
- `git pull` — скачать изменения с удалённого репозитория и слить с локальной веткой
- `git status` — посмотреть, что изменилось
- `git log` — история коммитов

**Работа с ветками:**
- `git branch` — список веток
- `git branch <name>` — создать ветку
- `git checkout <branch>` — переключиться на ветку
- `git checkout -b <branch>` — создать и переключиться
- `git merge <branch>` — влить ветку в текущую

**Когда изменения уже есть, но нужно временно отложить:**
- `git stash` — спрятать незакоммиченные изменения
- `git stash list` — посмотреть список спрятанных изменений
- `git stash pop` — вернуть последнее спрятанное и удалить из stash
- `git stash apply stash@{2}` — вернуть конкретное спрятанное (указывается номер)

**Отмена изменений:**
- `git revert <commit>` — создаёт новый коммит, который отменяет изменения указанного коммита. Безопасно для общей истории.
- `git reset` — откатывает историю, удаляя коммиты:
  - `--soft` — коммиты отменяются, изменения остаются в индексе (уже `git add`)
  - `--mixed` (по умолчанию) — коммиты отменяются, изменения остаются в рабочей директории (как после редактирования, но без `git add`)
  - `--hard` — коммиты и изменения удаляются полностью (⚠️ опасно)

**Работа с историей (продвинутое):**
- `git rebase <branch>` — переносит коммиты текущей ветки на конец другой. Делает историю линейной.
- `git cherry-pick <hash>` — берёт конкретный коммит из другой ветки и применяет в текущей. Нужен, например, чтобы перенести в `master` срочный баг-фикс, который уже есть в ветке отдельным коммитом, не вливая всю ветку целиком.

**Пример конфликта:**

- Есть ветка `master`
- Два разработчика создали ветки `feature/1` и `feature/2`
- Оба изменили одну и ту же строку в файле
- Первый смерджил свою ветку в `master` — всё ок
- Второй создаёт Merge Request (MR) / Pull Request (PR), но система сообщает о конфликте

**Что делает второй разработчик:**

1. Переключиться на `master` и обновить её:  
   `git checkout master`  
   `git pull`

2. Вернуться в свою ветку:  
   `git checkout feature/2`

3. Смерджить актуальный `master` в свою ветку:  
   `git merge master`  
   Git сообщит о конфликте и покажет файлы с маркерами:

       <<<<<<< HEAD
       твой код
       =======
       код из мастера
       >>>>>>> master

4. Вручную отредактировать файлы, оставив нужный результат (или объединив оба изменения), убрать маркеры

5. Сохранить файлы, сделать `git add` и завершить merge:  
   `git add .`  
   `git commit` (или `git merge --continue`)

6. Запушить ветку: `git push`

После этого MR/PR можно смерджить — конфликтов больше нет.

**CI/CD (Continuous Integration / Continuous Delivery)** — практика автоматизации сборки, тестирования и доставки кода.

- **CI (Continuous Integration)** — разработчики часто сливают код в основную ветку, после каждого слияния автоматически запускается сборка и тесты. Позволяет рано находить конфликты и ошибки.
- **CD (Continuous Delivery)** — код после прохождения CI автоматически разворачивается на тестовом окружении и готов к выкатке в прод (иногда выкатывается автоматически — Continuous Deployment).

**Как это выглядит на практике (GitLab CI):**
- Конфигурация в файле `.gitlab-ci.yml` в корне репозитория
- Состоит из **этапов (stages)**: обычно `build`, `test`, `deploy`
- Каждый этап может содержать несколько **джоб (jobs)**, которые выполняются параллельно

Пример простого `.gitlab-ci.yml`:

    stages:
      - test
      - build
      - deploy

    test:
      stage: test
      script:
        - pytest

    build:
      stage: build
      script:
        - docker build -t myapp .

    deploy:
      stage: deploy
      script:
        - kubectl apply -f deployment.yaml
      only:
        - main

**Почему test перед build:** тесты обычно запускаются в легковесном образе с Python, а сборка образа может требовать docker-in-docker или других тяжёлых инструментов. Если тесты падают — сборка даже не начинается, экономятся ресурсы.

**GitHub Actions** — аналог GitLab CI. Конфигурация в `.github/workflows/*.yml`.

**Возможности:**
- Автоматический прогон тестов и линтеров на каждый коммит
- Сборка артефактов (образов, бинарников)
- Релиз в разные окружения (dev/stage/prod)
- Безопасность (проверка зависимостей, секреты)

### Масштабирование

- **Вертикальное (scale up):** увеличить мощность существующего узла (сервера, ВМ, контейнера) — добавить CPU, RAM, диск. Применимо к любому компоненту, который работает на одном узле: БД, приложение, кэш.
- **Горизонтальное (scale out):** добавить больше экземпляров приложения (инстансов, процессов, контейнеров, серверов) и распределить нагрузку между ними. Требует балансировщика и управления состоянием (сессии, файлы, БД — выносятся вовне).

### Docker

**Что такое:** платформа для упаковки приложений в контейнеры — изолированные окружения с всем необходимым для работы.

**Базовые понятия:**
- **Образ (image)** — шаблон для создания контейнера (read-only)
- **Контейнер (container)** — запущенный экземпляр образа
- **Dockerfile** — инструкция по сборке образа
- **docker-compose** — запуск нескольких контейнеров сразу (для разработки)

**Пример Dockerfile:**
    FROM python:3.11-slim
    WORKDIR /app
    COPY requirements.txt .
    RUN pip install -r requirements.txt
    COPY . .
    CMD ["uvicorn", "main:app", "--host", "0.0.0.0"]

**Основные команды:**
- `docker build -t myapp .` — собрать образ
- `docker run myapp` — запустить контейнер
- `docker ps` — список запущенных контейнеров
- `docker compose up` — поднять всё из docker-compose.yml

**Образы Python (чем отличаются):**
- `python:3.11` — полный образ на Debian (много системных пакетов, компиляторы)
- `python:3.11-slim` — минимальный Debian + Python (рекомендуется)
- `python:3.11-alpine` — на базе Alpine Linux (ещё меньше, но проблемы с wheel'ами)

**Что такое wheel:** готовая сборка пакета. Для alpine нужна пересборка, т.к. он использует musl вместо glibc.

### Docker Swarm

**Что такое:** встроенная в Docker система оркестрации контейнеров. Позволяет объединить несколько серверов в кластер и запускать на них контейнеры.

**Когда использовать:** когда нужно управлять контейнерами на нескольких серверах, но не хочется сложностей Kubernetes. Проще в настройке, но уступает k8s в возможностях.

**Основные понятия:**
- **Manager** — управляет кластером, хранит состояние
- **Worker** — выполняет контейнеры
- **Service** — описание того, что и сколько запускать
- **Stack** — набор сервисов (аналог docker-compose в swarm)

**Чем отличается от Kubernetes:**
- Проще (одна команда `docker swarm init` — и кластер готов)
- Меньше возможностей (нет автоматического масштабирования по нагрузке, слабее управление сетью)
- Подходит для небольших проектов (до десятка серверов)

**Основные команды:**
- `docker swarm init` — инициализировать кластер
- `docker swarm join` — добавить узел в кластер
- `docker service create` — запустить сервис
- `docker stack deploy` — развернуть стек из docker-compose.yml

### Kubernetes (k8s)

Система для оркестрации контейнеров: автоматический деплой, масштабирование, управление. Позволяет декларативно описывать инфраструктуру (YAML-манифесты).

**Базовые объекты:**
- **Pod** — минимальная единица (один или несколько контейнеров)
- **Deployment** — описывает, сколько экземпляров пода должно работать
- **Service** — точка доступа к подам (балансировка)
- **ConfigMap / Secret** — хранение конфигурации
- **Ingress** — вход извне в кластер (HTTP-маршрутизация)

**Чем помогает:**
- Масштабирование (`kubectl scale`)
- Самоисцеление (перезапуск упавших подов)
- Обновление без простоя (rolling update)
- Хранение секретов

## 12. Тестирование

### Виды тестов

- **Юнит-тесты** — проверяют один модуль/функцию изолированно. Быстрые.
- **Интеграционные тесты** — проверяют взаимодействие модулей (например, с БД). Медленнее.
- **E2E (End-to-End)** — проверяют всю систему целиком (через UI или API). Самые медленные и хрупкие.

### Инструменты

- **pytest** — самый популярный фреймворк. Умеет фикстуры, параметризацию, плагины.
- **unittest** — встроенный в Python, менее удобный.

### Фикстуры (pytest)

Объекты, которые подготавливаются для тестов. Уровень действия (scope) определяет, как часто создаётся фикстура:
- `function` (по умолчанию) — для каждого теста
- `class` — для каждого класса тестов
- `module` — для каждого модуля
- `package` — для каждого пакета
- `session` — один раз за всю сессию тестирования

### Моки (unittest.mock)

Подменяют реальные объекты, чтобы изолировать тест. Вместо настоящего вызова — возвращается заранее заданный результат.

**Пример:** тестируем роут FastAPI, который вызывает внешний API

    # main.py
    import httpx
    from fastapi import FastAPI

    app = FastAPI()

    @app.get("/user/{user_id}")
    async def get_user(user_id: int):
        async with httpx.AsyncClient() as client:
            response = await client.get(f"https://ext-api.com/users/{user_id}")
            return response.json()

**Фикстура для тестового клиента** (scope="session" — создаётся один раз и используется во всех тестах):

    import pytest
    from fastapi.testclient import TestClient
    from main import app

    @pytest.fixture(scope="session")
    def client():
        return TestClient(app)

**Тест без мока** — реальный запрос к внешнему API:

    def test_get_user_without_mock(client):
        response = client.get("/user/123")
        # ❌ медленно, зависит от сети, внешний API может быть недоступен

**Тест с моком** — подменяем httpx, чтобы реального вызова не было:

    from unittest.mock import patch, Mock

    def test_get_user_with_mock(client):
        with patch("httpx.AsyncClient.get") as mock_get:
            mock_response = Mock()
            mock_response.json.return_value = {"id": 123, "name": "Test User"}
            mock_get.return_value = mock_response
            
            response = client.get("/user/123")
            
            assert response.status_code == 200
            assert response.json() == {"id": 123, "name": "Test User"}
            mock_get.assert_called_once_with("https://ext-api.com/users/123")

**Что изменилось:**
- Без мока: тест зависит от внешнего API (медленно, ненадёжно)
- С моком: тест изолирован, быстрый, проверяет только логику роута
- Мок проверяет, что роут делает правильный запрос к внешнему API

## 13. Мониторинг и логирование

### ELK Stack (Elasticsearch, Logstash, Kibana)

Стек для работы с данными: от сбора до визуализации.

- **Elasticsearch** — хранилище и поисковый движок. Любые данные (логи, документы, метрики) можно:
  - Хранить и искать по ним (полнотекстовый поиск)
  - Агрегировать и анализировать
  - Использовать как основную БД или как поисковый бэкенд для сайта

- **Logstash** — конвейер обработки данных. Забирает из источников (логи, БД, очереди), парсит, обогащает и отправляет в Elasticsearch.

- **Kibana** — интерфейс к Elasticsearch. Позволяет:
  - Строить дашборды и графики
  - Искать и фильтровать данные
  - Выполнять прямые запросы к Elasticsearch через Dev Tools

**Как логи из Python попадают в ELK:**

- **Через stdout (стандартный вывод)** — самый распространённый способ в контейнерах. Приложение пишет логи через `logger.info()` в stdout, а инфраструктура забирает их оттуда:
  - В Kubernetes: логи автоматически собираются агентами (Fluentd, Fluent Bit) и отправляются в Elasticsearch
  - В Docker: можно настроить драйвер логирования, который отправляет логи напрямую в Logstash или Elasticsearch

Это настраивается DevOps-ами, приложение ничего не знает про ELK — оно просто пишет в консоль.

## 14. Сети (для разработчика)

### Модели OSI и TCP/IP

**PDU (Protocol Data Unit)** — единица данных на каждом уровне. Название меняется при движении вниз по стеку.

**OSI модель (7 уровней):**

| Уровень | PDU | Примеры протоколов            | Примеры (где встречается) |
|---|---|-------------------------------|---|
| 7. Прикладной | Данные | HTTP, FTP, SMTP               | Код приложения (FastAPI, Django), Nginx как прокси, балансировщики слоя 7 |
| 6. Представления | Данные | TLS/SSL, JPEG, ASCII          | Шифрование/расшифровка в HTTPS, кодировка данных |
| 5. Сеансовый | Данные | NetBIOS, RPC                  | Управление сессиями, аутентификация |
| 4. Транспортный | Сегменты (TCP) / Датаграммы (UDP) | TCP, UDP                      | Балансировщики TCP, фаерволы (порты) |
| 3. Сетевой | Пакеты | IP, ICMP, OSPF                | Маршрутизаторы, фаерволы (IP-адреса) |
| 2. Канальный | Кадры (frames) | Ethernet, Wi-Fi (802.11), ARP | Коммутаторы, Wi-Fi точки, MAC-адреса |
| 1. Физический | Биты | Ethernet (физический), Wi-Fi (физический)           | Концентраторы, сетевые карты, кабели |

**TCP/IP модель (4 уровня, используется на практике):**

| Уровень OSI | Уровень TCP/IP | Протоколы |
|---|---|---|
| 5-7 (Прикладной, Представления, Сеансовый) | Прикладной | HTTP, HTTPS, FTP, SMTP, DNS, WebSockets |
| 4 (Транспортный) | Транспортный | TCP, UDP |
| 3 (Сетевой) | Сетевой | IP, ICMP, ARP |
| 1-2 (Канальный, Физический) | Канальный | Ethernet, Wi-Fi |

### Протоколы и стандарты

**Ключевые протоколы:**

- **TCP** — надёжный, с подтверждением доставки, порядок гарантирован (сайты, API)
- **UDP** — быстрый, без гарантий (видео, аудио, игры, DNS)
- **DNS** — превращает домены в IP
- **HTTP/1.1, HTTP/2, HTTP/3**:
  - HTTP/1.1 — запросы выполняются последовательно (один за другим). Работает поверх TCP.
  - HTTP/2 — запросы выполняются одновременно (мультиплексирование). Работает поверх TCP.
  - HTTP/3 — запросы выполняются одновременно (мультиплексирование). Работает поверх UDP, что даёт более быструю установку соединения и лучшую работу при плохой сети.
- **WebSocket** — поверх TCP, двусторонняя связь (чаты, уведомления)
- **TLS/SSL** — шифрование поверх TCP. TLS — современный протокол, SSL — устаревший протокол (но "SSL-сертификат" говорят до сих пор)

**Стандарты:**

- **RFC (Request for Comments)** — документы, в которых описаны все интернет-протоколы (TCP, IP, HTTP, DNS и т.д.). Каждый протокол имеет один или несколько RFC.

### HTTPS

**Как работает:**
- HTTP внутри шифрованного TLS-туннеля. Данные защищены от перехвата.
- Сертификат подтверждает, что сервер — это действительно тот, за кого себя выдаёт.

**Как настроить (два варианта):**

*На уровне инфраструктуры (делает DevOps):*
- Сертификат кладётся на Nginx (или другой прокси)
- Nginx принимает HTTPS, расшифровывает и проксирует на бэкенд по HTTP
- Приложение ничего не знает про HTTPS, работает как обычно

*На уровне приложения (делает разработчик):*
- Приложение само держит сертификат и работает по HTTPS
- В Uvicorn: `--ssl-keyfile=key.pem --ssl-certfile=cert.pem`
- Нужно, если нет прокси перед приложением (редко)

**Где брать сертификат:**
- Let's Encrypt (Certbot) — бесплатно, автоматически
- Самоподписанный — только для разработки
- Платные — для организаций с особыми требованиями

## 15. Другие технологии (для общего понимания)

- **Nginx:** веб-сервер, обратный прокси-сервер, балансировщик нагрузки.  
  *Обратный прокси* — принимает запросы из интернета и распределяет их на внутренние серверы (бэкенды). Задачи: SSL-терминация, раздача статики, балансировка, скрытие внутренней структуры.  
- **Amazon S3:** объектное хранилище в AWS. Данные хранятся как объекты (файл + метаданные + уникальный ID) в корзинах (buckets). Плоская структура (нет папок в классическом смысле).  
- **Apache Superset:** BI-инструмент для визуализации данных (дашборды, графики).  
- **Apache Hive:** хранилище данных поверх Hadoop.  
- **Apache Spark:** фреймворк для обработки больших данных.  
- **Postmark:** сервис для отправки транзакционных email.  
- **Apache Solr:** поисковая платформа на базе Lucene (как Elasticsearch).  
- **CDN (Content Delivery Network):** сеть доставки контента (например, Мегафон CDN). Кэширует статику (изображения, видео) на серверах по всему миру для ускорения загрузки.  
- **ElvisDam:** система управления цифровыми активами (DAM) для ресайза и реформата изображений. Не CDN, а скорее бэкенд для обработки медиа.
